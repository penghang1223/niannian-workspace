**目标：** 产出项目基础文档，经鉴微审查通过

| 步骤 | 负责 Agent | 具体操作 | 输出 |
|------|-----------|---------|------|

| 2b. 写角色卡 | 惊鸿✍️ | 5 个核心角色卡，每个包含：年龄、身份、性格、说话风格、金手指、核心动机、成长弧线、经典台词 | 角色卡.md |

**鉴微 PRD 审查清单：**
- [ ] 大纲三幕式结构完整（起承转合）
- [ ] 每章有明确的爽点和钩子
- [ ] 5 个角色人设不重复、有辨识度
- [ ] 主角动机清晰、成长弧线明确

- [ ] 目标平台适配

**通过标准：** 鉴微确认无 P0 问题

### 阶段3：试水章

**目标：** 写第 1 章，确认方向正确

| 步骤 | 负责 Agent | 具体操作 | 输出 |
|------|-----------|---------|------|
| 3a. 写第1章 | 惊鸿✍️ | 3000-4000 字，写到 chapters/第01章-{标题}.md | 第1章正文 |

| 3c. 判断 | 鉴微🛡️ | 方向OK → 阶段4；方向不行 → 惊鸿调整 → 重新试水 | 通过/重试 |

**试水章审稿重点（方向审，不做全书体检）：**
- [ ] 前 500 字抓力
- [ ] 核心设定清晰度
- [ ] 题材调性是否准
- [ ] 第一章钩子是否成立
- [ ] 主角是否让人想继续跟
- [ ] 有没有明显退稿风险

**试水章输出：**
- 本章评分
- 方向判断（成立/不成立）
- 最大亮点 / 最大问题
- 退稿风险
- 是否建议继续写
- 后续创作警戒线

### 阶段6：集中审稿（三位一体）

**目标：** 全书交付级审稿

**审稿配置（三位一体）：**
1. **鉴微的固有能力**：发展性编辑 + 人物审查 + 设定结构 + 行文编辑 + 盐选专项
2. **鉴微的 Skill**：editorial-review v2.1（六编辑体系）
3. **虚拟读者 v3.0**：30 位读者反馈 + 跨章记忆 + 角色/时间线/伏笔追踪

| 步骤 | 负责 Agent | 具体操作 | 输出 |
|------|-----------|---------|------|
| 6a. 读全文 | 鉴微🛡️ | 连读 12 章，建立基础约束（大纲/人设/世界观） | 全局认知 |
| 6b. 虚拟读者层 | 鉴微🛡️ | 调用虚拟读者 v3.0：30 位读者反馈、完读率、记忆点、劝退点 | 读者数据 |
| 6c. 编辑层 | 鉴微🛡️ | 六编辑体系：A 普通读者 → B 网文编辑 → C 专业作家 → D 挑刺读者 → E 设定审查 → F 改稿编辑 | 六编辑意见 |
| 6d. 问题分级 | 鉴微🛡️ | 统一归类 P0/P1/P2 | 问题清单 |
| 6e. 改稿建议 | 鉴微🛡️ | 每个问题给出具体改法（修改前 → 修改后） | 改稿方案 |
| 6f. 跨章回看 | 鉴微🛡️ | 人设连续性、时间线、伏笔回收、设定漂移、情感线推进 | 跨章问题清单 |

**集中审稿输出：**
- 逐章审稿意见
- 全书总评
- P0/P1/P2 汇总清单
- 最大亮点 Top 3
- 最大问题 Top 3
- AI 味定位（具体到句子）
- 修改建议（含修改前后对比）
- 投稿建议（是否建议直接投 / 需要修改后投）
- 必要时给重构建议

**P0/P1/P2 定义：**
| 级别 | 说明 | 处理 |
|------|------|------|
| **P0** | 必须改，不改会伤结构/人设/过稿率 | 修改后才能投稿 |
| **P1** | 建议改，影响阅读体验/节奏/商业性 | 强烈建议修改 |
| **P2** | 优化项，偏行文与精修 | 可选修改 |

**终章额外规则（第12章单独验收）：**
- 第12章不能视为普通章节，必须单独检查
- 必须有具体场景，不能只做抽象总结
- 必须回收前文伏笔和情感线

- 必须单独通过验收后，才能进入归档

**铁律：**
- 严格按 P0 → P1 → P2 优先级修改
- 每个修改都要有"修改前 → 修改后"对比
- 修改后必须自检，不能引入新问题

## 📸 Checkpoint 机制

每完成一个子步骤，写入 `progress/checkpoint.json`：

```json
{
  "task": "任务名称",
  "step": "当前步骤",
  "completed": 已完成章节数,
  "total": 总章节数,
  "timestamp": "2026-04-09T10:00:00+08:00",
  "next": "下一步做什么"
}
```

**Gateway 重启恢复流程：**
1. 读取 `checkpoint.json`
2. 确认已完成的步骤
3. 从下一步继续，不重复

**创作场景 Checkpoint：**
- 每写完一章 → 写入
- 每完成一次审稿 → 更新
- 每次修改完成 → 更新
- 第12章完成后 → 额外写一次“终章专项验收 checkpoint”

## 🔄 自我评估+重试机制

每次完成任务后执行：

1. **检查成功标准**：文件存在？内容正确？语法无误？
2. ✅ **成功** → 继续下一步
3. ❌ **失败** → 自动重试（最多 3 次）

**重试策略：**
- 第 1 次失败：检查错误信息，修复后重试
- 第 2 次失败：换一种实现方式
- 第 3 次失败：**强制降级**，报告给年年

**失败报告格式：**
```
[失败] 任务：xxx | 已重试 3 次 | 错误：xxx | 建议：xxx
```

## 📢 汇报规范

每次给主人汇报小说进度前，必须先实际检查文件和状态，再分开汇报：

1. **本地状态**：章节文件是否存在、字数、最新修改时间
2. **飞书状态**：哪些文档已创建、是否已放入项目子目录
3. **审稿状态**：是否已审、评分、P0/P1/P2 是否已处理

**禁止：**
- 凭上一轮记忆或旧检查结果直接汇报
- 把“本地写完”说成“项目已完成”
- 把“已派发任务”说成“已完成修改”
