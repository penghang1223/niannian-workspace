## [2026-03-27] 1. Pandas 基础

### 核心概念
- **DataFrame**：Pandas 的核心二维数据结构，类似 Excel 表格或 SQL 表，由行索引和列索引组成，每列可为不同数据类型
- **Series**：一维数组结构，DataFrame 的一列即为一个 Series
- **Index**：行/列的标签标识，支持自定义索引和默认整数索引
- **数据对齐**：Pandas 自动按索引对齐数据，类似数据库的 JOIN

### 关键知识点
1. **创建 DataFrame**
   - 从字典：`pd.DataFrame({'col1': [1,2], 'col2': [3,4]})`
   - 从列表：`pd.DataFrame([[1,2],[3,4]], columns=['A','B'])`
   - 从 NumPy 数组、CSV、Excel 等外部文件读取

2. **索引与筛选**
   - `loc[]`：按标签索引，如 `df.loc[0, 'name']`
   - `iloc[]`：按位置索引，如 `df.iloc[0:3, 1:2]`
   - 布尔索引：`df[df['age'] > 30]`
   - `query()` 方法：`df.query('age > 30 and city == "北京"')`

3. **数据操作**
   - `head(n)` / `tail(n)`：查看前/后 n 行
   - `info()` / `describe()`：查看数据结构和统计信息
   - `sort_values(by='col')`：排序
   - `dropna()` / `fillna(value)`：缺失值处理
   - `drop_duplicates()`：去重

4. **聚合与分组**
   - `groupby()`：按列分组，支持 `mean()`/`sum()`/`count()`/`agg()`
   - `agg()` 多聚合：`df.groupby('city').agg({'salary': ['mean','max'], 'age': 'count'})`
   - `pivot_table()`：创建透视表，类似 Excel 数据透视表
   - `transform()`：在分组内做变换，保持原 DataFrame 形状

5. **数据合并**
   - `merge()`：类似 SQL JOIN，支持 inner/left/right/outer
   - `concat()`：按行或按列拼接多个 DataFrame

### 实践要点
- 望舒日常用 Pandas 分析用户行为数据、销售数据、A/B 测试结果
- `groupby` + `agg` 是最常用组合，快速生成用户分群统计
- `pivot_table` 快速生成多维报表，做数据趋势分析
- 注意缺失值处理（NaN），分析前务必用 `info()` 检查数据质量

## [2026-03-27] 2. NumPy 基础

### 核心概念
- **ndarray**：NumPy 的核心 N 维数组对象，支持同类型元素的高效存储和运算
- **向量化运算**：无需循环，直接对整个数组做数学运算，底层用 C 实现，速度极快
- **广播（Broadcasting）**：不同形状数组之间的运算规则，小数组自动扩展匹配大数组
- **轴（Axis）**：数组的维度方向，axis=0 沿行（列方向运算），axis=1 沿列（行方向运算）

### 关键知识点
1. **创建数组**
   - `np.array([1,2,3])`：从列表创建
   - `np.zeros((3,4))`：全零矩阵
   - `np.ones((2,3))`：全一矩阵
   - `np.arange(0, 10, 2)`：等差序列 [0,2,4,6,8]
   - `np.linspace(0, 1, 5)`：等分序列
   - `np.random.randn(3,3)`：随机正态分布矩阵
   - `np.eye(3)`：单位矩阵

2. **数组属性**
   - `shape`：维度形状 (行,列)
   - `dtype`：数据类型（int64, float64 等）
   - `ndim`：维度数
   - `size`：元素总数

3. **索引与切片**
   - 基本切片：`arr[0:3, 1:4]`（行 0-2，列 1-3）
   - 布尔索引：`arr[arr > 5]`
   - 花式索引：`arr[[0,2,4], :]`（取特定行）
   - ⚠️ 切片是视图（view），修改会影响原数组；需 copy 用 `arr.copy()`

4. **数学运算**
   - 元素运算：`+`, `-`, `*`, `/`, `**`
   - 矩阵运算：`np.dot(A, B)` 或 `A @ B`（矩阵乘法）
   - 聚合函数：`np.sum()`, `np.mean()`, `np.std()`, `np.max()`, `np.min()`
   - 支持 axis 参数指定方向：`arr.sum(axis=0)` 按列求和

5. **广播规则**
   - 规则1：维度不同，小维度前面补 1
   - 规则2：对应维度要么相同，要么其中一个为 1
   - 示例：(3,4) 的数组 + (4,) 的数组 → 后者自动广播为 (1,4) → (3,4)
   - 实用场景：数据标准化 `(x - mean) / std`

6. **数组变形**
   - `reshape(m, n)`：改变形状
   - `flatten()`：展平为一维
   - `transpose()` 或 `.T`：转置
   - `concatenate()` / `vstack()` / `hstack()`：拼接

### 实践要点
- Pandas 底层依赖 NumPy，理解 NumPy 能更好地优化数据处理性能
- 向量化运算比 Python for 循环快 100 倍以上，数据分析必备
- 广播是数据标准化、归一化的核心机制
- 望舒用 NumPy 主要做快速数值计算和随机模拟（如蒙特卡洛模拟）

## [2026-03-27] 3. Matplotlib 基础

### 核心概念
- **Figure**：画布，所有绘图元素的顶层容器
- **Axes**：子图/坐标系，实际绑图的区域，一个 Figure 可包含多个 Axes
- **pyplot**：Matplotlib 的状态机接口（`plt.xxx`），最常用的快速绑图方式
- **Seaborn**：基于 Matplotlib 的高级统计可视化库，图表更美观

### 关键知识点
1. **折线图（Line Chart）**
   ```python
   import matplotlib.pyplot as plt
   plt.plot(x, y, color='blue', linewidth=2, linestyle='--', marker='o')
   plt.title('趋势图')
   plt.xlabel('时间')
   plt.ylabel('数值')
   plt.show()
   ```
   - 适用：时间序列、趋势分析（DAU/MAU 趋势）

2. **散点图（Scatter Plot）**
   ```python
   plt.scatter(x, y, s=area, c=colors, alpha=0.5)
   ```
   - 适用：两个变量的相关性分析（如广告投入 vs 销售额）
   - 支持用 `s`（大小）和 `c`（颜色）编码第三/四维度

3. **柱状图（Bar Chart）**
   ```python
   plt.bar(categories, values, color='steelblue')
   plt.barh(categories, values)  # 横向柱状图
   ```
   - 适用：分类数据对比（各渠道用户量对比）
   - 分组柱状图：两次 bar() 调用，设置不同 x 位置

4. **饼图（Pie Chart）**
   ```python
   plt.pie(sizes, labels=labels, autopct='%1.1f%%', startangle=90)
   ```
   - 适用：占比分析（用户来源分布、市场份额）
   - ⚠️ 超过 7 个分类建议换用柱状图

5. **子图（Subplots）**
   ```python
   fig, axes = plt.subplots(2, 2, figsize=(12, 8))
   axes[0,0].plot(x, y1)  # 左上
   axes[0,1].bar(x, y2)   # 右上
   ```

6. **美化与导出**
   - `plt.style.use('ggplot')` / `seaborn` 主题
   - `plt.legend()` 显示图例
   - `plt.savefig('chart.png', dpi=300, bbox_inches='tight')` 导出高清图
   - 中文支持：`plt.rcParams['font.sans-serif'] = ['SimHei']`

### 实践要点
- 望舒应掌握 4 种基础图表：折线（趋势）、柱状（对比）、散点（关联）、饼图（占比）
- 汇报时优先用 Seaborn 或 Plotly，图表更美观、交互性更好
- 配色遵循品牌色或使用 ColorBrewer 等专业配色方案
- 图表标题和轴标签务必清晰，避免"图表裸奔"

## [2026-03-27] 4. SQL 进阶

### 核心概念
- **JOIN**：多表关联查询，将分散在不同表的数据组合在一起
- **子查询**：嵌套在其他查询中的 SELECT 语句，用于复杂条件过滤
- **窗口函数**：在不减少行数的情况下对数据进行分组计算（OVER 子句）
- **CTE（公用表表达式）**：用 WITH 定义临时命名结果集，提升可读性

### 关键知识点
1. **JOIN 类型**
   - `INNER JOIN`：只返回两表匹配的行
   - `LEFT JOIN`：左表全部 + 右表匹配（最常用）
   - `RIGHT JOIN`：右表全部 + 左表匹配
   - `FULL OUTER JOIN`：两表全部，不匹配填 NULL
   - `CROSS JOIN`：笛玄机积（慎用）
   - ⚠️ 望舒最常用 LEFT JOIN（如：用户表 LEFT JOIN 订单表）

2. **子查询**
   - WHERE 子查询：`WHERE id IN (SELECT user_id FROM orders WHERE amount > 100)`
   - FROM 子查询（派生表）：`SELECT * FROM (SELECT ...) AS sub`
   - EXISTS 子查询：比 IN 效率更高，适合大数据集
   - 关联子查询：子查询引用外层表的列

3. **窗口函数（Window Functions）**
   - `ROW_NUMBER()`：行号，常用于取每组第一条
   - `RANK()` / `DENSE_RANK()`：排名（有/无间隔）
   - `LAG()` / `LEAD()`：获取前/后 N 行的值（计算环比/同比）
   - `SUM() OVER (PARTITION BY ... ORDER BY ...)`：累计求和
   - `AVG() OVER (...)`：移动平均
   ```sql
   -- 示例：每个用户的订单累计金额
   SELECT user_id, order_date, amount,
          SUM(amount) OVER (PARTITION BY user_id ORDER BY order_date) AS running_total
   FROM orders
   ```

4. **CTE 公用表表达式**
   ```sql
   WITH active_users AS (
       SELECT user_id, COUNT(*) AS order_count
       FROM orders
       WHERE order_date >= '2026-01-01'
       GROUP BY user_id
       HAVING COUNT(*) >= 3
   )
   SELECT u.name, au.order_count
   FROM users u JOIN active_users au ON u.id = au.user_id
   ```

5. **实用技巧**
   - `CASE WHEN` 条件表达式：数据分桶、打标签
   - `HAVING`：分组后的条件过滤（区别于 WHERE）
   - `COALESCE()`：空值替换
   - `DATE_FORMAT()` / `EXTRACT()`：日期处理
   - `UNION` / `UNION ALL`：合并结果集

### 实践要点
- 望舒应该能写中等复杂度的 SQL，独立从数据库取数分析
- 窗口函数是进阶必备：做用户留存分析、漏斗分析、排名分析
- 避免 SELECT *，只取需要的字段，提升查询效率
- 大表 JOIN 注意索引优化，必要时用 EXPLAIN 查看执行计划
- 常用场景：用户分群查询、转化漏斗、留存率计算、RFM 分析

### 核心概念

- **维度 vs 度量**：维度是分类字段（如地区、产品类型），度量是数值字段（如销售额、数量）

### 关键知识点
1. **Tableau 核心操作**
   - **数据连接**：连接 MySQL/PostgreSQL/Excel/Google Sheets 等
   - **拖拽绑图**：维度拖到行/列，度量拖到行/列/标签/颜色
   - **图表类型选择**：
     - 折线图：时间趋势（月度 DAU）
     - 柱状图：分类对比（各渠道转化率）

   - **参数**：动态控制图表的输入值

   - **DAX 公式**：数据分析表达式，创建计算列和度量值

   - **关系模型**：建立多表之间的关系（星型模型/雪花模型）
   - **书签（Bookmarks）**：保存视图状态，制作交互式报告

3. **Tableau vs Power BI 对比**

   | 适合场景 | 大型企业、深度分析 | 中小企业、日常报表 |

   - **少即是多**：每张图只传达一个核心观点
   - **色彩克制**：不超过 5-7 种颜色，高亮关键数据点
   - **叙事逻辑**：仪表盘按"概览 → 细节 → 行动"的逻辑排列

## [2026-03-27] 6. A/B 测试

### 核心概念
- **A/B 测试**：将用户随机分为两组（对照组 A 和实验组 B），比较不同方案的效果差异
- **假设检验**：统计学方法，判断实验结果是否由随机波动导致
- **零假设（H₀）**：两个版本没有差异；备择假设（H₁）：存在差异
- **统计显著性**：p 值 < 0.05 表示差异有统计学意义（95% 置信度）

### 关键知识点
1. **实验设计四要素**
   - **基线转化率（Baseline）**：当前版本的转化率，如 5%
   - **最小可检测效应（MDE）**：你希望检测到的最小改进幅度（如相对提升 10%）
   - **统计功效（Power）**：检测到真实差异的概率，标准为 80%
   - **显著性水平（α）**：误报概率，标准为 5%（95% 置信度）

2. **样本量计算**
   - 公式：`n = 2 × (Zα/2 + Zβ)² × p̄(1-p̄) / (p₂ - p₁)²`
   - 影响因素：基线越低、MDE 越小 → 所需样本越大
   - 示例：基线 5%，MDE 10%（即检测到 5%→5.5%），约需每组 30,000 用户
   - 工具推荐：Optimizely Sample Size Calculator、Evan Miller Calculator

3. **结果分析方法**
   - **卡方检验**：最常用的转化率差异检验
   - **t 检验**：连续指标（如平均订单金额）的差异检验
   - **置信区间**：给出差异的范围估计，比单一 p 值更有信息量
   - **Bayesian 方法**：给出"变体更好的概率"，比频率派更直观

4. **常见错误**
   - ❌ **偷看效应**：每天看结果，看到显著就停止 → 误报率飙到 30%+
   - ❌ **样本量不足**：没提前算好就开测 → 无法得出可靠结论
   - ❌ **多重比较**：同时测多个指标不做校正 → 假阳性累积
   - ❌ **忽略用户分流**：同一用户看到不同版本 → 结果偏差
   - ❌ **幸存者偏差**：只分析留存用户，忽略流失用户

5. **实验平台**
   - 国外：Optimizely、VWO、Google Optimize（已停用）、Statsig
   - 国内：火山引擎（字节）、A/B Test 平台（阿里）、自建平台

### 实践要点
- A/B 测试是产品决策的核心武器，用数据代替直觉
- 测试前：写好假设、确定指标、算好样本量、确定测试时长
- 测试中：不要偷看！提前确定的时长到了再分析
- 测试后：不仅看 p 值，还要看效果大小和业务影响
- 望舒应建立"测试文化"：小改动也要测，积累数据驱动的决策经验

## [2026-03-27] 7. 用户画像

### 核心概念
- **用户画像（User Persona）**：基于真实数据构建的虚拟用户模型，代表某一类目标用户的典型特征
- **用户分群（User Segmentation）**：将用户按共同特征分成不同群体，每群体制定差异化策略
- **RFM 模型**：最近购买时间（Recency）、购买频率（Frequency）、消费金额（Monetary）
- **用户生命周期**：新用户 → 活跃用户 → 沉默用户 → 流失用户

### 关键知识点
1. **数据驱动的用户画像构建步骤**
   - Step 1：收集数据（行为数据 + 人口统计 + 心理特征）
   - Step 2：数据清洗和特征工程
   - Step 3：聚类分析（K-means、层次聚类）发现用户分群
   - Step 4：为每个群体命名、描述特征、写用户故事
   - Step 5：验证画像是否能指导产品决策

2. **画像维度**
   - **人口统计**：年龄、性别、地域、收入、职业
   - **行为数据**：使用频率、功能偏好、路径偏好、留存率
   - **心理特征**：动机、痛点、目标、价值观
   - **设备/渠道**：设备类型、操作系统、获客渠道

3. **RFM 模型实操**
   - R（最近购买）：最近一次消费距今天数 → 1-5 分
   - F（购买频率）：一定周期内购买次数 → 1-5 分
   - M（消费金额）：一定周期内总消费额 → 1-5 分
   - 组合分类：高价值客户（555）、流失预警（155）、新客户（511）等
   - 按 RFM 分数制定差异化运营策略

4. **用户生命周期分析**
   - **新增**：首次使用产品的用户
   - **活跃**：近期有核心行为的用户（如 DAU/WAU/MAU）
   - **留存**：在特定时间窗口后仍然活跃的用户（次日/7日/30日留存）
   - **沉默**：一段时间没有活跃但未达到流失阈值
   - **流失**：超过定义的时间窗口没有回访

5. **常用工具**
   - 数据分析：SQL + Python（Pandas/Sklearn 聚类）
   - 可视化：Tableau / Power BI（画像仪表盘）
   - 用户调研：问卷星、Typeform、用户访谈
   - 行为分析：Mixpanel、Amplitude、神策数据

### 实践要点
- 用户画像是产品决策的基础，"不为画像服务的产品设计是盲目的"
- 定期更新画像（每季度），用户特征会随时间和产品迭代变化
- 画像数量控制在 3-5 个，太多反而失去聚焦
- 画像应包含具体场景："小王，25岁，互联网运营，每天通勤1小时用手机学英语"
- 落地关键：画像必须转化为可执行的产品决策和运营策略

## [2026-03-27] 8. 需求分析

### 核心概念
- **用户故事（User Story）**：从用户视角描述需求的标准格式，"作为<角色>，我想要<功能>，以便<价值>"
- **需求优先级排序**：用结构化方法对需求进行排序，确保资源投入产出最大化
- **验收标准（Acceptance Criteria）**：需求的量化完成标准，用于开发自测和 QA 验收
- **MVP（最小可行产品）**：用最少的功能验证核心假设

### 关键知识点
1. **用户故事编写（INVEST 原则）**
   - **I**ndependent：独立的，不依赖其他故事
   - **N**egotiable：可协商的，细节可讨论
   - **V**aluable：有价值的，对用户有意义
   - **E**stimable：可估算的，开发能评估工作量
   - **S**mall：足够小，能在 1 个 Sprint 内完成
   - **T**estable：可测试的，有明确验收标准

2. **需求优先级排序框架**
   - **RICE 框架**（最推荐）
     - R = Reach（触达）：多少用户会受影响（如每季度 10000 用户）
     - I = Impact（影响）：对用户的影响程度（3=巨大, 2=高, 1=中, 0.5=低, 0.25=微弱）
     - C = Confidence（信心）：对估算的确信度（100%=高, 80%=中, 50%=低）
     - E = Effort（工作量）：人月数
     - 分数 = (R × I × C) / E

   - **Kano 模型**
     - 基本型需求：必须有，没有用户会不满
     - 期望型需求：越多越好，直接影响满意度
     - 兴奋型需求：超预期惊喜，差异化竞争关键

   - **MoSCoW 方法**
     - Must have：必须做
     - Should have：应该做
     - Could have：可以做
     - Won't have：暂不做

   - **价值/复杂度矩阵**
     - 高价值低复杂度 → 立即做
     - 高价值高复杂度 → 规划做
     - 低价值低复杂度 → 有空做
     - 低价值高复杂度 → 不做

3. **需求文档结构**
   - 背景与目标
   - 用户故事 & 验收标准
   - 功能描述 & 交互原型
   - 数据埋点需求
   - 异常处理 & 边界情况
   - 排期 & 里程碑

4. **需求来源与收集**
   - 用户反馈（客服、社区、用户访谈）
   - 数据分析（漏斗流失点、功能使用率）
   - 竞品分析（竞品新功能、行业趋势）
   - 业务方需求（销售、运营、老板）
   - 技术需求（架构升级、性能优化）

### 实践要点
- 望舒的核心能力是"定义正确的事"，需求分析是起点
- 每个需求都要回答"为什么做"和"做了之后怎么衡量成功"
- 需求评审时用 RICE 量化排序，避免"谁声音大谁赢"
- 需求文档要清晰到"开发者不需要再找你确认"
- 定期回顾需求池，清理过时需求，保持 Backlog 整洁

## [2026-03-27] 9. 用户访谈

### 核心概念
- **用户访谈**：一对一定性研究方法，通过对话深入了解用户需求、行为和痛点
- **结构化访谈**：预设固定问题，按顺序提问，适合定量验证
- **半结构化访谈**：有大纲但灵活追问，最常用
- **非结构化访谈**：自由对话，适合探索性研究

### 关键知识点
1. **访谈准备**
   - 明确研究目标（要验证什么假设？要发现什么未知？）
   - 制定访谈大纲（5-8 个核心问题 + 追问点）
   - 招募合适用户（5-8 人即可发现 80% 的问题）
   - 准备录音/录像设备（征得用户同意）

2. **访谈技巧**
   - **开放式提问**：用"什么/怎么/为什么"开头，避免是非题
     - ❌ "你觉得这个功能好用吗？"
     - ✅ "你上次用这个功能时，发生了什么？"
   - **5 Whys 追问法**：连续追问 5 个"为什么"，找到根本原因
   - **具体化追问**：让用户描述具体场景，而非泛泛而谈
     - "能给我讲讲你上次遇到这个问题的具体经过吗？"
   - **沉默技巧**：问完后等 3-5 秒，给用户思考空间
   - **不引导**：避免暗示"正确答案"
     - ❌ "你是不是觉得这个按钮应该更大？"
     - ✅ "你在这个页面上第一眼看到的是什么？"

3. **问卷设计原则**
   - 问题从简单到复杂排列
   - 避免双重否定和专业术语
   - 每题只问一件事
   - 提供"不确定/不适用"选项
   - 控制在 5-10 分钟完成
   - 先放筛选题，再放核心题

4. **问卷题型**
   - 单选题：互斥选项
   - 多选题：可多选
   - 李克特量表：1-5 分或 1-7 分满意度
   - NPS（净推荐值）：0-10 分推荐意愿
   - 开放题：收集定性反馈

5. **访谈分析**
   - 转录访谈记录
   - 用亲和图法（Affinity Diagram）归类主题
   - 提取关键洞察和用户引言
   - 与定量数据交叉验证
   - 输出用户故事和机会点

### 实践要点
- 用户访谈是望舒的"超能力"，数据告诉你"是什么"，访谈告诉你"为什么"
- 每月至少做 3-5 次用户访谈，保持对用户的敏感度
- 访谈时多听少说（比例 80:20），你的任务是倾听不是推销
- 不要问用户"想要什么功能"，而要问"遇到了什么问题"
- 访谈洞察要形成可执行的行动项，不只是写报告

## [2026-03-27] 10. 竞品分析

### 核心概念
- **竞品分析**：系统性地研究竞争对手的产品、策略、优劣势，为自身产品决策提供参考
- **直接竞品**：解决同样问题、面向同样用户的产品
- **间接竞品**：用不同方式解决同样问题的产品
- **替代方案**：用户可能选择的非竞品解决方案（如 Excel 替代 SaaS 工具）

### 关键知识点
1. **竞品分析框架**
   - **波特五力模型**
     - 现有竞争者的威胁
     - 潜在进入者的威胁
     - 替代品的威胁
     - 供应商的议价能力
     - 买方的议价能力

   - **SWOT 分析**
     - Strengths（优势）：我方做得好的地方
     - Weaknesses（劣势）：需要改进的地方
     - Opportunities（机会）：市场机会
     - Threats（威胁）：竞争和外部风险

2. **竞品分析维度**
   - **产品功能**：核心功能对比矩阵、特色功能、功能完整度
   - **用户体验**：交互设计、上手难度、信息架构
   - **商业模式**：定价策略、盈利模式、免费策略
   - **用户增长**：获客渠道、增长策略、市场活动
   - **技术架构**：技术栈、性能、开放 API
   - **团队与融资**：团队背景、融资轮次、估值

3. **信息收集渠道**
   - 产品体验：亲自注册使用竞品（最重要的方法！）
   - 官方渠道：官网、帮助中心、更新日志、定价页
   - 第三方数据：SimilarWeb、App Annie、七麦数据
   - 社交媒体：知乎、Twitter、Product Hunt、即刻
   - 用户反馈：竞品的 App Store 评论、客服评价
   - 行业报告：艾瑞、易观、Gartner

4. **差异化策略**
   - **功能差异化**：做竞品没有的功能
   - **体验差异化**：更简洁/更快/更好用
   - **价格差异化**：更低价格或更高性价比
   - **服务差异化**：更好的售后/客户成功
   - **生态差异化**：更强的集成和生态
   - **垂直差异化**：深耕某个细分场景

5. **输出模板**
   | 维度 | 我方 | 竞品A | 竞品B |
   |------|------|-------|-------|
   | 目标用户 | | | |
   | 核心功能 | | | |
   | 定价 | | | |
   | 用户规模 | | | |
   | 优势 | | | |
   | 劣势 | | | |

### 实践要点
- 竞品分析不是一次性工作，要建立持续监控机制
- 重点不是"抄竞品"，而是"理解市场和用户选择"
- 定期（每月）更新竞品分析，关注竞品动态
- 分析完竞品后，关键是回答"我们的差异化策略是什么"
- 避免"功能军备竞赛"：不盲目追赶竞品功能，聚焦核心价值

## [2026-03-27] 11. PRD 撰写

### 核心概念
- **PRD（Product Requirements Document）**：产品需求文档，描述产品"做什么"和"怎么做"的核心文档
- **MRD（Market Requirements Document）**：市场需求文档，描述"为什么做"和市场机会
- **BRD（Business Requirements Document）**：商业需求文档，描述商业价值和 ROI
- **原型（Prototype）**：产品的可视化表达，从低保真线框图到高保真交互原型

### 关键知识点
1. **PRD 文档结构（标准模板）**
   - **文档信息**：产品名称、版本号、作者、日期、状态
   - **背景与目标**：为什么做这个功能，解决什么问题
   - **用户故事**：As a [role], I want [feature], so that [benefit]
   - **功能描述**：详细功能说明、交互流程、状态机
   - **验收标准（AC）**：给定/当/则格式，开发自测和 QA 验收依据
   - **数据埋点**：关键事件和属性定义
   - **异常处理**：错误情况和边界条件
   - **非功能性需求**：性能、安全、兼容性
   - **排期与里程碑**：时间线和交付节点

2. **好的 PRD 特征**
   - **无歧义**：任何人读完理解一致
   - **可测试**：每个需求都有明确的验收标准
   - **完整性**：覆盖正常流程、异常流程、边界情况
   - **可追溯**：需求有来源、有优先级、有负责人
   - **简洁**：去掉冗余信息，直达核心

3. **交互流程描述方法**
   - **用户流程图**：用泳道图描述用户操作路径
   - **状态机**：描述系统状态流转（如订单状态）
   - **用例图**：UML 标准，描述角色与功能的关系
   - **时序图**：描述前端/后端/第三方系统的交互时序

4. **AI 产品 PRD 特殊考量**
   - **输入/输出定义**：明确模型的输入格式和期望输出
   - **准确率要求**：定义可接受的准确率/召回率基线
   - **兜底策略**：模型不确定时的降级方案
   - **数据需求**：训练数据来源、标注标准、数据量
   - **反馈闭环**：用户反馈如何回流到模型优化
   - **安全与合规**：内容审核、隐私保护、偏见控制

5. **常用工具**
   - 文档：飞书文档、Notion、Confluence
   - 原型：Figma、Sketch、Axure
   - 流程图：draw.io、ProcessOn、Excalidraw
   - 协作：飞书多维表格（需求跟踪）

### 实践要点
- PRD 不是"写给自己看的笔记"，而是团队协作的契约
- 先画原型再写 PRD，"一图胜千言"
- PRD 要在评审会上过一遍，确认所有人理解一致
- 需求变更是常态，但必须走变更流程并更新文档
- AI 产品的 PRD 要特别关注"模型做不到怎么办"的兜底方案

## [2026-03-27] 12. AI 产品架构

### 核心概念
- **LLM（大语言模型）**：基于 Transformer 的大规模预训练语言模型（GPT、Claude、Qwen 等）
- **Agent（智能体）**：能自主规划、使用工具、完成任务的 AI 系统
- **RAG（检索增强生成）**：结合外部知识库检索和 LLM 生成，解决知识时效性和幻觉问题
- **向量数据库**：存储和检索高维向量的专用数据库（Pinecone、Milvus、Weaviate）

### 关键知识点
1. **LLM 应用架构层次**
   - **应用层**：面向用户的 AI 产品（ChatGPT、Copilot、AI 客服）
   - **编排层**：Prompt 管理、Agent 框架、工作流编排（LangChain、Dify、Coze）
   - **模型层**：LLM API 调用（OpenAI、Anthropic、国内大模型）
   - **数据层**：向量数据库、知识库、用户数据存储

2. **Agent 架构**
   - **感知**：接收用户输入（文本/语音/图像）
   - **规划**：将复杂任务拆解为子任务（ReAct、CoT 推理）
   - **工具调用**：调用外部 API/数据库/搜索引擎
   - **记忆**：短期记忆（对话上下文）+ 长期记忆（向量存储）
   - **执行**：调用工具、生成结果
   - **反思**：验证结果、迭代优化

3. **RAG 工作流程**
   - Step 1：文档切分（Chunking）→ 将长文档切为语义片段
   - Step 2：向量化（Embedding）→ 用 Embedding 模型转为向量
   - Step 3：存储索引 → 存入向量数据库
   - Step 4：检索 → 用户查询转为向量，检索最相关片段
   - Step 5：生成 → 将检索结果 + 用户问题一起给 LLM 生成答案

4. **向量数据库对比**
   | 数据库 | 特点 | 适用场景 |
   |--------|------|----------|
   | Pinecone | 全托管、易用 | 快速上线、小团队 |
   | Milvus | 开源、高性能 | 大规模、私有部署 |
   | Weaviate | 多模态、GraphQL | 复杂查询 |
   | Chroma | 轻量、嵌入式 | 开发测试、小项目 |
   | Qdrant | Rust 实现、高效 | 高性能需求 |

5. **关键设计决策**
   - 模型选择：效果 vs 成本 vs 延迟的权衡
   - Prompt Engineering：系统提示词设计、Few-shot 示例
   - 上下文窗口：控制 token 消耗和信息密度
   - 幻觉控制：RAG + 检索 + 事实校验
   - 安全防护：输入过滤、输出审核、权限控制

6. **常见 AI 产品形态**
   - ChatBot：对话式交互（客服、助手）
   - Copilot：嵌入式辅助（代码、写作、设计）
   - Agent：自主完成任务（自动化工作流）
   - 工具型：单一功能 AI（翻译、摘要、生图）

### 实践要点
- 望舒不需要懂模型训练，但要理解模型能力和局限
- RAG 是企业 AI 落地的主流方案，解决"模型不知道公司数据"的问题
- Agent 是未来方向，但目前仍面临可靠性、成本、安全的挑战
- 设计 AI 产品时，关键问题是"用户愿意为哪些 AI 能力付费"
- 持续关注 AI 领域发展：模型能力在快速进化，产品形态也在不断演变

## [2026-03-27] 13. 产品路线图

### 核心概念
- **产品路线图（Product Roadmap）**：产品未来发展方向和计划的可视化表达
- **Now-Next-Later**：一种灵活的路线图格式（现在做/下一步做/以后做）
- **主题型路线图**：围绕战略主题（而非具体功能）展示产品方向
- **OKR 对齐**：路线图中的每个里程碑对应可衡量的目标和关键结果

### 关键知识点
1. **路线图类型**
   - **时间线型**：按时间排列，有具体里程碑和交付日期
   - **Now-Next-Later 型**：无具体时间，按优先级分三档
   - **主题型**：围绕战略目标组织（如"提升留存""扩展市场"）
   - **功能型**：按功能模块或产品线组织（适合大型产品）

2. **路线图制定流程**
   - Step 1：回顾现状 — 当前产品数据、用户反馈、技术债务
   - Step 2：明确愿景 — 产品 1-3 年后要达到什么状态
   - Step 3：拆解目标 — 将愿景拆为可衡量的 OKR
   - Step 4：识别机会 — 用户需求、市场机会、技术趋势
   - Step 5：排序优先级 — 用 RICE/Kano 框架排序
   - Step 6：估算资源 — 开发资源、时间周期
   - Step 7：评审对齐 — 与团队和利益相关方对齐
   - Step 8：持续迭代 — 每月/季度更新

3. **版本迭代策略**
   - **MVP → MVE → MLP**
     - MVP（最小可行产品）：验证核心假设
     - MVE（最小可爱产品）：提升体验到用户愿意推荐
     - MLP（最小可规模化产品）：具备扩展能力
   - **双轨开发**：探索轨（验证新想法）+ 交付轨（稳定发布）
   - **版本节奏**：小版本频繁发布（2 周一个 Sprint），大版本季度规划

4. **利益相关方沟通**
   - **高层/投资人**：关注战略目标、ROI、市场机会
   - **研发团队**：关注技术可行性、排期、依赖关系
   - **销售/运营**：关注功能上线时间、客户承诺
   - **用户**：关注新功能预告（公开路线图）

5. **常用工具**
   - 产品管理：Jira、Linear、飞书多维表格
   - 路线图可视化：Productboard、Aha!、Notion
   - 项目管理：飞书项目、Asana、Monday

### 实践要点
- 路线图不是"承诺"，而是"当前最佳判断"，保持灵活
- 向上沟通用"为什么"（战略价值），向下沟通用"怎么做"（技术方案）
- 避免"功能堆积"：路线图要讲一个连贯的故事
- 每个版本聚焦 1-2 个核心目标，不要面面俱到
- 定期回顾路线图，根据数据和市场变化调整优先级

## [2026-03-27] 14. 项目管理（敏捷开发/Scrum）

### 核心概念
- **敏捷开发（Agile）**：迭代式、增量式的软件开发方法，强调快速响应变化
- **Scrum**：最流行的敏捷框架，以 Sprint 为周期进行迭代开发
- **Sprint**：固定时间周期（通常 2 周），完成一组选定的用户故事
- **看板（Kanban）**：另一种敏捷方法，基于"拉动式"工作流，无固定迭代

### 关键知识点
1. **Scrum 核心角色**
   - **Product Owner（产品负责人）**：管理产品 Backlog，定义优先级，代表用户利益
   - **Scrum Master**：确保 Scrum 流程执行，移除团队障碍
   - **开发团队**：跨职能团队（前端/后端/QA/设计），自组织完成工作

2. **Scrum 核心事件**
   - **Sprint 计划会**：从 Backlog 中选出本次 Sprint 要完成的用户故事
   - **每日站会**：每人 15 秒说"昨天做了什么/今天做什么/有什么阻碍"
   - **Sprint 评审会**：向利益相关方演示完成的功能
   - **Sprint 回顾会**：团队复盘，讨论改进点
   - **Backlog 梳理会**：细化即将到来的需求，拆分大故事

3. **核心工件**
   - **产品 Backlog**：所有待做需求的有序列表，PO 负责维护
   - **Sprint Backlog**：本次 Sprint 要完成的故事和任务
   - **增量（Increment）**：Sprint 结束时交付的可用产品增量
   - **用户故事卡片**：故事描述 + 验收标准 + 估点

4. **估算方法**
   - **故事点（Story Points）**：用斐波那契数列（1,2,3,5,8,13）估算复杂度
   - **计划扑克（Planning Poker）**：团队成员各自出牌估算，差异大时讨论
   - **T 恤尺码**：XS/S/M/L/XL 快速粗估
   - **速率（Velocity）**：团队每 Sprint 完成的平均故事点，用于预测排期

5. **看板方法**
   - 列：待办 → 进行中 → 待测试 → 已完成
   - **WIP 限制**：每列限制同时进行的任务数（如"进行中"最多 3 个）
   - **拉动式**：完成后主动拉取下一个任务，而非被动分配
   - 适合：运维、支持、需求不固定的团队

6. **望舒在敏捷中的角色**
   - 管理和优先排序产品 Backlog
   - 写好用户故事和验收标准
   - 参加 Sprint 计划会和评审会
   - 及时回答开发团队的问题
   - 在 Sprint 中不做需求变更（保护开发专注度）

7. **常见误区**
   - ❌ 把站会开成"汇报会"（应聚焦障碍）
   - ❌ Sprint 中途频繁插入需求（应尊重 Sprint 承诺）
   - ❌ 只追求速度不看质量（要平衡速度和质量）
   - ❌ 忽视回顾会（持续改进是敏捷的精髓）
   - ❌ 把 Scrum 当教条（应该根据团队实际情况调整）

### 实践要点
- 敏捷不是"不要计划"，而是"拥抱变化的同时保持纪律"
- 望舒是团队的"翻译官"：把业务语言翻译成开发语言
- Sprint 节奏感很重要：固定周期、固定仪式、可预测交付
- 持续改进：每个回顾会至少落实 1-2 个改进点
- 透明沟通：用看板/日报让所有人对进度有清晰认知
