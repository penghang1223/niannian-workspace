# 版本控制规范

> 记忆系统版本管理标准 | 版本：v1.0

---

## 📋 版本号格式

```
v{major}.{minor}[.{patch}]
```

| 部分 | 说明 | 示例 |
|------|------|------|
| **major** | 重大变更（内容重构、主题变化） | v1.0 → v2.0 |
| **minor** | 小幅更新（内容补充、修正） | v1.0 → v1.1 |
| **patch** | 微小修正（错别字、格式） | v1.0.0 → v1.0.1 |

---

## 📝 版本变更标准

### Major 更新（v1.0 → v2.0）

触发条件：
- 记忆主题发生重大变化
- 内容结构重构
- 核心信息变更（如主人偏好完全改变）
- 合并多个记忆项

示例：
```
主人之前偏好："我喜欢详细回答"
主人现在偏好："我偏好简洁风格，详细反而不喜欢"
→ 内容核心变化，v1.0 → v2.0
```

---

### Minor 更新（v1.0 → v1.1）

触发条件：
- 内容补充（添加新细节）
- 小幅修正（更新部分信息）
- 添加新的关联链接
- 格式优化

示例：
```
在"沟通风格"中添加新的偏好细节
→ 内容补充，v1.0 → v1.1
```

---

### Patch 更新（v1.0.0 → v1.0.1）

触发条件：
- 错别字修正
- 格式微调
- 链接修复
- 元数据更新

示例：
```
修正错别字或更新链接
→ 微小修正，v1.0.0 → v1.0.1
```

---

每个记忆项顶部必须包含：

# 记忆标题

> 类型：{type} | 创建：{YYYY-MM-DD} | 版本：v{major}.{minor}

---

## 📦 归档规范

### 何时归档

- 信息过时且不再相关（超过 3 个月）
- 被新记忆完全替代
- 临时性任务已完成
- 主人明确要求删除

### 归档方式

**方式一：添加 .archived 后缀**
```
v1.0 → v1.0.archived
```

**方式二：移动到归档目录**
```
items/preferences/旧偏好.md → archive/preferences/旧偏好.md
```

**方式三：添加归档标记**
```markdown
> ⚠️ 已归档：2026-06-02 | 原因：信息过时
```

### 归档模板

```markdown
# {记忆标题}

> ⚠️ **已归档** | 归档日期：{YYYY-MM-DD} | 原因：{原因}

---

## 📝 归档说明

{说明为什么归档，是否有替代记忆}

---

## 📝 历史版本

{保留原有变更历史}
```

## 🔗 版本追溯

### 查看变更历史

每个记忆项的变更历史表格应包含：

| 列 | 说明 |
|------|------|
| **版本** | 版本号（v1.0, v1.1 等） |
| **日期** | 变更日期（YYYY-MM-DD） |
| **变更内容** | 简要描述变更内容 |
| **操作者** | 年年 / 主人 |
| **原因** | 变更原因（可选） |

---

### 版本对比

当需要对比两个版本时：

```bash
# 年年命令示例
年年，对比 {记忆名} 的 v1.0 和 v1.1 版本
```

年年应输出：
- 两个版本的完整内容
- 差异高亮显示
- 变更原因说明

## 💡 最佳实践

### ✅ 应该做的

1. **每次更新都记录** - 不遗漏任何变更
2. **版本号准确** - 根据变更类型选择正确的版本增量
3. **变更描述清晰** - 让未来的人能理解变更原因
4. **保留历史** - 不删除旧版本记录
5. **及时归档** - 过时信息及时标记

### ❌ 应该避免的

1. **沉默更新** - 不记录变更历史
2. **版本跳跃** - 无重大变更却跳 major 版本
3. **描述模糊** - "更新内容"这样的描述
4. **删除历史** - 抹去变更记录
5. **从不归档** - 记忆系统越来越臃肿
