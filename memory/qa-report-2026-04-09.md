## 2️⃣ Redis Cache-Aside 实现审查

### 审查的 4 个缓存端点

| 端点 | TTL | 缓存 Key 策略 | 失效触发 | 评级 |
|------|-----|--------------|---------|------|
| `GET /api/stats` | 30s | `stats:public` | 注册成功时 `invalidateCache('stats:*')` | ✅ |
| `GET /api/recent-registrations` | 60s | `recent:{limit}` | 注册成功时 `invalidateCache('recent:*')` | ✅ |
| `GET /api/registrations-by-location` | 120s | `location:{lat}:{lng}:{rad}:{lim}` | 未失效（合理，注册不直接影响区域查询） | ✅ |
| `GET /api/admin/stats` | 30s | `stats:admin` | 未自动失效（⚠️ 见问题） | ⚠️ |

### 审查发现

**✅ 优点：**
- Cache-Aside 模式正确实现（先读缓存 → miss 再查 DB → 写缓存）
- 缓存辅助函数有完整的 try-catch 错误处理
- 注册成功后主动失效 stats/recent 缓存
- TTL 分级合理（30s/60s/120s 随数据变化频率递增）
- 缓存未命中时优雅降级（返回 DB 数据）

**⚠️ 问题（2 项）：**

**P1-1: admin/stats 缓存未失效**
- 注册成功后只失效 `stats:*` 和 `recent:*`
- `stats:admin` 不在失效范围内
- **影响**: 新注册后 admin 统计可能显示旧数据最多 30 秒
- **建议**: 在 `/api/register` 的 invalidateCache 调用中加上 `invalidateCache('stats:admin')`

**P1-2: Redis 连接失败无降级**
- `redisClient.connect()` 在启动时调用但无错误处理
- 如果 Redis 不可用，`getCache`/`setCache` 会 try-catch 返回 null/静默失败
- **实际影响**: 低（缓存 miss 会走 DB，功能不受影响）
- **建议**: 启动时检测 Redis 连接，失败时给出明确警告日志
