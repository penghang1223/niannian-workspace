## 一、现状诊断

### 🔍 现有 5 种文件类型分析

| 类型 | 示例文件 | 特征 | 问题 |
|------|---------|------|------|
| **A. 知识笔记型** | 霓裳/望舒/鉴微 | 长篇技术教程，包含概念+代码示例 | 根本不是"教训"，是学习参考资料 |
| **B. 知识摘录型** | 年年 lessons.md (2026-03-28) | "从某篇文章/某人学到X" | 来自外部输入，缺乏自己的实践验证 |
| **C. 结构化学习型** | 玄机 dev_engineer | 有来源/价值评级/应用场景/待验证 | 最接近目标格式，但仍然偏"学习"而非"解决问题" |
| **D. 平台知识型** | main (旧版) | "核心概念→关键知识点→实践要点"三段式 | 知道什么，但没有实际用过 |
| **E. 空缺型** | 部分 agents | 文件不存在或几乎为空 | 无内容 |

### ❌ 六大核心问题

```
问题 1: 学习 ≠ 教训
─────────────────────
大多数文件记录的是"我学了什么"，而非"我踩了什么坑、怎么爬出来的"。
真正的 lesson 来自于实践中的困难和解决过程。

问题 2: 概念堆砌，无可执行性
─────────────────────
"核心概念→关键知识点→实践要点"是百科全书格式，
不是操作手册。读完后不知道"具体第一步做什么"。

问题 3: 缺少问题-方案因果链
─────────────────────
没有"因为遇到了X问题，所以用了Y方法，得到了Z结果"的完整链条。
大部分只是"X知识很好，以后可以用"。

问题 4: 没有成功标准
─────────────────────
"预期改善"写得模糊（如"效率提升"），没有量化指标或可验证的结果。

问题 5: 缺少失败/踩坑记录
─────────────────────
几乎全是"学到了好东西"，没有"我犯了什么错、怎么发现的、怎么纠正的"。
失败经验的复用价值远高于成功经验。

问题 6: 不可复用
─────────────────────
其他 Agent 读了之后，不知道该怎么"直接照做"。
缺少：具体命令、精确参数、环境前提、注意事项。
```

## 二、Lesson 质量定义

### 什么是好 Lesson？

```
好 Lesson = 我用了 [具体方法] 解决了 [具体问题]
           步骤是 A → B → C
           成功标准是 [可验证的结果]
           其他 Agent 能直接照做
```

### 四维评估标准

| 维度 | 定义 | 评分标准 |
|------|------|----------|
| **🔧 工具型** | 能写成代码/脚本/skill | 5=可直接执行的代码/命令 3=有伪代码或命令片段 1=纯文字描述 |
| **🎯 可操作** | 有具体步骤和示例 | 5=步骤编号+完整示例 3=步骤概述+部分示例 1=方向性建议 |
| **✅ 可验证** | 有明确的成功标准 | 5=量化指标+验证命令 3=定性标准 1="效果应该不错" |
| **♻️ 可复用** | 其他 Agent 能直接应用 | 5=换个人/场景直接用 3=需要适配但有方向 1=高度依赖上下文 |

**总分 = 四维之和，满分 20**

| 总分 | 等级 | 处理 |
|------|------|------|
| 16-20 | ⭐ 优秀 | 直接保留 |
| 11-15 | 🟡 良好 | 补充具体细节 |
| 6-10 | 🟠 合格 | 需要重写结构 |
| 1-5 | 🔴 不合格 | 删除或完全重写 |

## 三、Lesson 模板

### 基础模板（所有 Agent 通用）

```markdown
## [日期] [简短标题]

**问题**: [遇到了什么具体问题？描述现象和影响]
**方法**: [用了什么具体方法/工具解决？]
**来源**: [从哪里学到的？(可选)]

### 具体步骤
1. [第一步：做什么，怎么做]
2. [第二步：做什么，怎么做]
3. [第三步：做什么，怎么做]

### 验证方式
- [如何确认问题已解决？]
- [具体命令/指标/标准]

### 适用范围
- ✅ 适用：[什么场景下用这个方法]
- ❌ 不适用：[什么场景下不该用]

### 反馈记录
- [ ] 已验证：[日期] [结果]
- [ ] 待验证：[预期效果]
```

### 高级模板（含踩坑/失败经验）

```markdown
## [日期] 🔴 踩坑：[简短标题]

**现象**: [具体表现是什么？]
**根因**: [为什么会发生？]
**错误做法**: [之前怎么做的（导致了问题）]
**正确做法**: [后来怎么改的（解决了问题）]

### 具体修复步骤
```bash
# 步骤 1: [说明]
具体命令

# 步骤 2: [说明]
具体命令
```

### 预防措施
- [以后怎么避免这个问题再次发生？]
- [有没有自动检测的方式？]

### 复用价值
- 适用 Agent：[哪些角色会遇到类似问题]
- 触发条件：[什么情况下应该回看这条 lesson]
```

### 代码型模板（技术 Agent 专用）

```markdown
## [日期] ⚡ 代码：[简短标题]

**问题**: [代码/系统层面的具体问题]
**解决方案**: [一段可直接复用的代码/配置]

### 代码
```[语言]
// 可直接复制使用的代码
```

### 使用说明
- 文件位置：[放在哪里]
- 前置条件：[需要什么环境/依赖]
- 调用方式：[怎么用]

### 效果对比
- 优化前：[数据/表现]
- 优化后：[数据/表现]
```

## 四、改进指南

### Step 1: 分类现有内容

对每个 Agent 的 lessons.md 执行分类：

```
对每一条记录，判断：
├─ 是"学到了知识"→ 标记为 📚 学习笔记（移到参考资料区）
├─ 是"解决了问题"→ 标记为 💡 教训（保留在主区）
├─ 是"踩了坑"→ 标记为 🔴 踩坑（保留在主区，高优先级）
└─ 是"通用知识"→ 标记为 📖 参考（移到 SHARED_KNOWLEDGE.md）
```

### Step 2: 重写结构

新 lessons.md 结构：

```markdown
# [Agent名] 经验记录

> 最后更新：[日期]
> 维护者：[Agent名]

---

## 💡 实战教训（核心区）

[按时间倒序排列，最近的在最前面]

### [日期] [标题]
[用模板填写]

---

## 🔴 踩坑记录（高优先区）

[只记录失败/问题经验]

---

## 📚 学习笔记（参考资料区）

[精简后的知识摘录，每条不超过 5 行要点]

---

## 📊 效果追踪

| 日期 | 教训标题 | 验证状态 | 实际效果 |
|------|---------|---------|---------|
| | | ✅已验证/⏳待验证/❌无效 | |
```

### Step 3: 质量检查清单

每条 lesson 写完后，用这个清单自检：

```
□ 标题是否清晰？（读标题就知道解决了什么问题）
□ 是否有具体问题描述？（不是"学习了X"，是"遇到了X问题"）
□ 是否有可执行的步骤？（其他 Agent 能照着做）
□ 是否有验证方式？（怎么知道成功了）
□ 是否有适用范围？（什么时候用，什么时候不用）
□ 是否已实际使用过？（不是"我觉得可以用"，是"我用过并且有效"）
□ 是否有反馈记录？（实际效果如何）
```

### Step 4: 写作规范

| 规范 | 说明 | 示例 |
|------|------|------|
| **动词优先** | 用动词开头，描述行动 | ✅ "用 `rg` 搜索替代 `grep`" ❌ "搜索很重要" |
| **数字说话** | 量化效果 | ✅ "编译时间从 45s 降到 12s" ❌ "编译快了很多" |
| **命令具体** | 给出完整命令 | ✅ `pytest -k "test_login" --cov=app` ❌ "用 pytest 跑测试" |
| **场景明确** | 写清什么时候用 | ✅ "文件超过 500 行时" ❌ "大型文件" |
| **一次一事** | 每条 lesson 只解决一个问题 | 一个 lesson = 一个可执行的知识点 |

## 五、示例对比

### ❌ 差的 Lesson（现有典型）

```markdown
## React Hooks

### 核心概念
Hooks 是 React 16.8 引入的特性...

### 关键知识点
- useState 管理状态
- useEffect 管理副作用
- useContext 共享状态

### 实践要点
- 单一职责
- 依赖数组完整
- 避免过优化
```

**评分：工具型 1 | 可操作 1 | 可验证 1 | 可复用 1 = 4分 🔴**

> 这是百科全书，不是教训。不知道遇到了什么问题、怎么解决的。

### ✅ 好的 Lesson

```markdown
## [2026-03-30] useEffect 内存泄漏：异步请求未取消

**问题**: 组件卸载后 setState 仍被调用，控制台报 "Can't perform a React state update on an unmounted component"
**方法**: 在 useEffect 清理函数中取消未完成的请求

### 具体步骤
1. 在 useEffect 内部声明 cancelled 标志
2. 在 fetch 的 .then 中检查 cancelled
3. cleanup 函数中设置 cancelled = true

### 代码
```jsx
useEffect(() => {
  let cancelled = false;
  fetch(`/api/users/${userId}`)
    .then(res => res.json())
    .then(data => {
      if (!cancelled) setUser(data);
    });
  return () => { cancelled = true; };
}, [userId]);
```

### 验证方式
- 切换页面时控制台无 "unmounted component" 警告
- Network 面板中取消的请求显示 cancelled

### 适用范围
- ✅ 有异步请求的 useEffect
- ✅ 组件可能频繁挂载/卸载
- ❌ 同步操作不需要
- ❌ 使用 AbortController 的场景（更推荐用 AbortController）

### 反馈记录
- ✅ 已验证 2026-03-30：Dashboard 页面切换时警告消除
```

**评分：工具型 5 | 可操作 5 | 可验证 4 | 可复用 5 = 19分 ⭐**

> 有具体问题、完整代码、验证方式、适用边界。其他 Agent 遇到同样问题能直接照做。

## 六、Agent 特定指南

### 各角色应记录什么类型的 Lesson

| Agent | 应重点记录 | 示例 |
|-------|-----------|------|
| **main (年年)** | 协调踩坑、工作流优化、工具使用技巧 | "子 Agent 超时怎么处理" |
| **dev_engineer (玄机)** | 代码问题、性能优化、架构决策 | "TypeScript 编译慢的解决方法" |
| **frontend_dev (霓裳)** | UI 问题、CSS 坑、组件优化 | "Flexbox 在 Safari 的兼容问题" |
| **qa_engineer (鉴微)** | 测试失败案例、自动化技巧 | "Playwright 定位器选择策略" |
| **product_manager (望舒)** | 需求变更教训、沟通技巧 | "需求评审中被挑战的应对方法" |
| **taiyi (太一)** | 研究发现的可执行结论 | "高效搜索技巧" |
| **lingxi/灵犀** | 创意工作的方法论 | "文案 A/B 测试的有效方法" |
| **jinghong/惊鸿** | 写作技巧、叙事结构 | "角色塑造的具体手法" |

### 迁移指南：学习笔记 → 实战教训

对于已有的"百科全书"型 lessons.md：

```
原内容处理策略：
1. 完整教程 → 保留为独立的参考资料文件（如 refs/react-hooks.md）
2. 知识要点 → 压缩为 3-5 行关键点，放入"学习笔记区"
3. 实践经验 → 提取为独立 lesson，用新模板重写
4. 空洞总结 → 删除（"X 很重要" 没有价值）
```

## 七、运维机制

### 定期审查（每两周一次）

由年年或指定 Agent 执行：

```
审查项：
1. 是否有超过 30 天未验证的"待验证" lesson？→ 催促验证或降级
2. 是否有评分 < 11 分的 lesson？→ 标记重写
3. 是否有重复/冲突的 lesson？→ 合并或标注差异
4. 新增 lesson 数量是否合理？（过少=没记录，过多=没筛选）
```

### 知识流动

```
Agent lessons.md → 年年审查 → SHARED_KNOWLEDGE.md（高复用价值的）
                         → MEMORY.md（核心经验）
                         → AGENTS.md 规则（反复踩的坑→变成规则）
```

### 淘汰机制

```
连续 2 次审查未验证的 lesson → 降级为"学习笔记"
连续 3 次标记"无效"的方法 → 删除
评分持续 < 6 分的 Agent lessons.md → 触发年年介入辅导
```

## 八、快速参考

### 一句话标准

> **好的 lesson 是"处方"，不是"教科书"。**
> 其他 Agent 读了能直接照做，做完了能验证是否成功。

### 自检口诀

```
有没有问题？（问题）
怎么解决的？（方法）
第一步干啥？（步骤）
成功长啥样？（验证）
别人能用吗？（复用）
```

### 评分速查

```
每个维度问一个问题：
🔧 工具型：能写成一行命令吗？
🎯 可操作：读完知道第一步做什么吗？
✅ 可验证：有办法验证成功了吗？
♻️ 可复用：换个人能直接用吗？
```
