## 1. 产品定位与目标用户

### 1.1 产品定位

Dashboard v4.0 是多 Agent 团队的**统一指挥中心**，为主人（团队管理者）提供对 13 个 Agent 的**实时可视化监控、任务调度、知识沉淀和团队效能分析**能力。

从 v3 的"信息展示面板"升级为**可操作的管理终端**——不仅看到状态，还能直接干预和调度。

### 1.2 目标用户

| 用户角色 | 核心诉求 | 使用频率 |
|---------|---------|---------|
| **主人（航航）** — 主要用户 | 全局掌控团队状态、快速调度任务、发现异常 | 每日多次 |
| **年年（协调员）** — 次要用户 | 看到任务分发结果、Agent 响应状态 | 任务分发时 |
| **各 Agent** — 被管理对象 | 被监控状态、接收任务指令 | 间接参与 |

### 1.3 核心价值主张

> **一句话**：一个面板看清 13 个 Agent 的一切——谁在忙、谁在闲、任务到哪了、出了什么问题。

### 1.4 痛点分析（基于 v3 反馈）

| 痛点 | v3 现状 | v4.0 解决方案 |
|------|--------|-------------|
| 离线状态不准 | Dashboard v3 经常进程挂掉不自知 | 健康检查 + 自动重启 + 心跳机制 |
| 只能看不能做 | 纯展示，无法操作 | 内置任务调度、Agent 指令下发 |
| 无历史趋势 | 只有实时快照 | 时间序列数据 + 趋势图表 |
| 无告警机制 | 异常需要人工发现 | 智能告警 + 飞书通知 |
| 移动端不可用 | 仅桌面端 | 响应式设计 + 飞书卡片通知 |

### 模块详情

#### M1: 总览面板（Overview）
- **全局状态仪表盘**：13 个 Agent 在线/离线/忙碌/空闲状态一览
- **关键指标卡片**：今日任务完成数、成功率、平均响应时间、活跃 Agent 数
- **告警中心**：异常事件实时展示（Agent 离线、任务超时、错误率飙升）
- **最新动态流**：时间线展示最近 24h 内的所有重要事件（任务完成、Agent 状态变化等）

#### M2: Agent 管理（Agent Manager）
- **状态卡片矩阵**：13 张 Agent 卡片，每张显示头像、角色、状态、当前任务、最后活跃时间
- **Agent 详情页**：点击卡片展开，显示能力评分、历史任务列表、专属配置（SOUL.md 摘要、模型路由）
- **模型路由可视化**：当前各 Agent 使用的模型，支持手动切换
- **工作区浏览**：每个 Agent 的 workspace 目录文件浏览（只读）

#### M3: 任务管理（Task Manager）
- **看板视图**：TODO / IN_PROGRESS / DONE / BLOCKED 四列看板
- **波次执行视图**：GSD 工作流的波次可视化（Wave 1 → Wave 2 → ...）
- **任务创建/编辑**：在面板内直接创建任务、分配 Agent、设置优先级
- **任务详情**：验收标准（AC）、产出物、时间线、关联文档

#### M4: 效能分析（Analytics）
- **团队效能仪表盘**：任务完成率趋势、Agent 负载分布、平均处理时间
- **Agent 排行榜**：按完成任务数、成功率、响应速度排名
- **模型消耗统计**：各模型的 token 使用量和预估成本
- **工作流健康度**：GSD 各步骤的耗时分布、瓶颈识别

#### M5: 系统设置（Settings）
- **告警规则配置**：Agent 离线超过 N 分钟告警、任务超时告警等
- **通知渠道**：飞书私信/群聊通知开关
- **主题配置**：暗色/亮色主题切换
- **数据源配置**：STATE.yaml 路径、workspace 路径、心跳间隔等

## 3. 用户故事 / 用户旅程

### 用户故事

#### 3.1 主人日常巡检（Daily Standup）

```
作为 主人（团队管理者），
我想要 打开 Dashboard 就一眼看到所有 Agent 的状态和今天要做的事，
以便 快速了解团队全貌，决定今天重点推进什么。

验收标准：
✅ 打开页面 < 2 秒加载完成
✅ 13 个 Agent 状态一目了然（彩色编码：绿=在线、红=离线、黄=忙碌、灰=空闲）
✅ 关键指标（任务完成率、今日任务数）置顶展示
✅ 有异常（如 Agent 离线）时红色高亮 + 弹窗提示
```

#### 3.2 快速调度任务

```
作为 主人，
我想要 在 Dashboard 上直接给某个 Agent 发任务，
以便 不用切到飞书群或命令行就能完成调度。

验收标准：
✅ 点击 Agent 卡片 → 弹出操作面板 → 输入任务描述 → 确认发送
✅ 任务发送后实时更新看板（新增一条 TODO）
✅ 可设置优先级（P0/P1/P2）和截止时间
✅ 任务发送后飞书通知主人确认
```

#### 3.3 异常告警

```
作为 主人，
我想要 当 Agent 离线或任务超时时立即收到通知，
以便 第一时间介入处理，不让问题发酵。

验收标准：
✅ Agent 连续 10 分钟无心跳 → Dashboard 红色告警 + 飞书推送
✅ 任务超过预计时间 2 倍未完成 → 超时告警
✅ 告警可一键静默（snooze）或标记已处理
✅ 告警历史可追溯
```

#### 3.4 复盘分析

```
作为 主人，
我想要 看到过去一周团队的工作效能数据，
以便 在周会上有数据支撑地复盘和决策。

验收标准：
✅ 支持按天/周/月切换时间范围
✅ 展示任务完成趋势折线图
✅ Agent 负载热力图（谁最忙、谁最闲）
✅ 关键指标可导出为图片/表格
```

### 用户旅程（主人一天的使用路径）

```
08:00  🌅 打开 Dashboard → 总览面板 → 查看隔夜活动摘要
       → 发现 dev_engineer 昨晚无任务执行 → 确认是否需要安排

09:00  📊 查看晨会数据 → Agent 全部在线 → 确认今日任务分布

10:00  📋 收到主人任务 → 在 Dashboard 创建任务
       → 分配 Agent → 看到波次自动规划

14:00  🔍 检查任务进度 → 波次执行视图 → 发现某 Agent 响应超时
       → 点击催促 → 飞书自动发送催促消息

17:00  📈 查看今日完成情况 → 效能分析 → 确认完成率

22:00  🌙 晚间检查 → 确认所有任务状态 → 设定隔夜任务
```

## 附录

### A. 技术栈建议

| 层 | 推荐方案 | 理由 |
|---|---------|------|
| 前端 | React 18 + TypeScript + Vite | v3 已有 React 基础，TypeScript 保证质量 |
| UI 库 | Tailwind CSS + Headless UI | 轻量、暗色主题开箱即用 |
| 图表 | Recharts | React 生态，轻量够用 |
| 后端 | Node.js + Express | 轻量，与 OpenClaw 生态一致 |
| 实时更新 | Server-Sent Events (SSE) | 比 WebSocket 简单，单向推送足够 |
| 数据源 | STATE.yaml + 文件系统 | 简单可靠，无需数据库 |

### B. 与 v3 的差异对比

| 维度 | v3 | v4.0 |
|------|-----|------|
| 架构 | 单体 JS + 静态页面 | React SPA + Node.js API |
| 数据 | 硬编码 + 一次读取 | 文件监听 + SSE 实时推送 |
| 交互 | 只读展示 | 可操作（创建任务、调度 Agent） |
| 告警 | 无 | 离线检测 + 超时告警 + 飞书通知 |
| 分析 | 无 | 效能仪表盘 + 趋势图 |
| 主题 | 仅暗色 | 暗色/亮色可切换 |
| 健康检查 | 无 | 自身心跳 + 自动重启 |

### C. 术语表

| 术语 | 含义 |
|------|------|
| Agent | OpenClaw 多 Agent 团队中的一个智能体 |
| GSD | Get Shit Done，6 步标准化工作流 |
| 波次（Wave） | 任务并行执行的分组 |
| AC | Acceptance Criteria，验收标准 |
| SSE | Server-Sent Events，服务端推送 |
| STATE.yaml | 团队任务状态的单一数据源 |
| SOUL.md | Agent 的人格/角色定义文件 |
