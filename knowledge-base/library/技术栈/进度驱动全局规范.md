## ⚠️ 核心原则

### ❌ 禁止使用的时间驱动表达
- "运行 30 分钟"
- "等待 5 分钟"
- "每小时执行一次"
- "24 小时后汇报"
- "timeout 600s"
- "heartbeat every 30m"

### ✅ 必须使用的进度驱动表达
- "完成 10 个任务"
- "执行 100 次查询"
- "处理 50 条消息"
- "收集 1000 次阅读"
- "跑满 100 次基准测试"
- "完成 1 个波次"

---

## 🎯 为什么禁止时间驱动？

### 1. 结果不可比
```
❌ 时间驱动：
- 第 1 轮：30 分钟，处理 8 个任务
- 第 2 轮：30 分钟，处理 12 个任务
→ 样本量不同，无法比较

✅ 进度驱动：
- 第 1 轮：完成 10 个任务
- 第 2 轮：完成 10 个任务
→ 样本量相同，可以比较
```

### 2. 浪费时间
```
❌ 时间驱动：
- 任务 5 分钟完成了，还要等 25 分钟
- 或者 30 分钟到了，任务还没跑完

✅ 进度驱动：
- 任务完成就结束，立即开始下一轮
- 确保每次都跑满样本量
```

### 3. 统计不显著
```
❌ 时间驱动：
- 30 分钟只跑了 3 次，样本不足
- 结果偶然性大

✅ 进度驱动：
- 必须跑满 100 次，确保统计显著性
- 结果可信
```

## 🔧 改造指南

### Step 1：识别时间驱动
```markdown
❌ 找到以下表达：
- "X 分钟"
- "X 小时"
- "timeout Xs"
- "every Xm/Xh"
- "等待 X 时间"
```

### Step 2：转换为进度驱动
```markdown
✅ 替换为以下表达：
- "完成 X 个任务"
- "执行 X 次操作"
- "处理 X 条消息"
- "收集 X 个样本"
- "跑满 X 次测试"
```

### Step 3：验证
```markdown
检查标准：
- 不再出现时间单位（分钟/小时/秒）
- 所有任务有明确的完成标准
- 样本量可比较
```

## 📊 进度驱动配置模板

### Agent 心跳
```json
// ❌ 时间驱动
{
  "heartbeat": {
    "every": "30m"
  }
}

// ✅ 进度驱动
{
  "heartbeat": {
    "trigger": "per_message_count",
    "threshold": 50
  }
}
```

### 实验循环
```json
// ❌ 时间驱动
{
  "experiment": {
    "duration": "30m",
    "timeout": "600s"
  }
}

// ✅ 进度驱动
{
  "experiment": {
    "completion_condition": "task_count >= 10",
    "min_samples": 100
  }
}
```

### Cron 任务
```json
// ❌ 时间驱动
{
  "schedule": "0 */2 * * *"
}

// ✅ 进度驱动
{
  "trigger": "queue_depth >= 50",
  "batch_size": 10
}
```
