# 🤖 OpenClaw 多 Agent 协作自主优化指南

> 版本：v1.0  
> 创建时间：2026-04-08  
> 灵感来源：Karpathy Autoresearch  
> 适用：OpenClaw 多 Agent 系统

---

## 🎯 目标

**改进 Agent 协作规则以提升任务完成率/响应时间**

---

## 📋 核心规则

### 1. 每次只改一个参数
- ✅ 超时设置（timeout）
- ✅ 心跳间隔（heartbeat）
- ✅ 并发数（maxConcurrent）
- ✅ 通信方式（sessions_send vs feishu_im）
- ✅ 重试次数（retryCount）
- ✅ 模型路由（primary/fallback）

**禁止**：一次改多个参数（无法归因）

---

### 2. 记录改动原因
每次改动前必须写：
```markdown
## 实验记录 #001

**改动内容**：timeout 从 300s 改为 600s

**改动原因**：
- 连续 4 次 09:00 指标分析 timeout
- 太一代码实战连续 3 次 timeout
- 需要更长时间完成复杂任务

**预期效果**：timeout 错误减少 80%+
```

---

### 3. 运行测试（进度驱动）
- 启动 Agent 协作流程
- **完成 10 个任务** 或 **处理 50 条消息**
- 收集指标数据

---

### 4. 数据驱动决策
| 指标 | 决策 |
|------|------|
| 任务完成率提升≥20% | ✅ 保留，继续优化 |
| 任务完成率提升 10-20% | 🟡 保留，观察长期 |
| 任务完成率变化<10% | ❌ 回滚，换方向 |
| 任务完成率下降 | ❌ 立即回滚，记录教训 |

---

## 🔄 实验循环（进度驱动）

```
1. 读取当前配置（openclaw.json）
   ↓
2. 分析数据（timeout 错误/任务完成率）
   ↓
3. 提出假设（"如果改 X，完成率会提升"）
   ↓
4. 修改配置（只改一个参数）
   ↓
5. 运行测试（完成 10 个任务或 50 条消息）
   ↓
6. 收集指标
   ↓
7. 决策（保留/回滚）
   ↓
8. 记录实验日志
   ↓
9. 重复循环
```

## 📊 实验日志模板

```markdown
## 实验 #001

**日期**：2026-04-08

**改动内容**：timeout 从 300s 改为 600s

**改动原因**：
- 连续 4 次 09:00 指标分析 timeout

**预期效果**：timeout 错误减少 80%+

**实际效果**：
- timeout 错误：5 次 → 1 次（-80%）✅
- 任务完成率：75% → 88%（+17.3%）✅
- 平均响应时间：120s → 180s（+50%）⚠️

**决策**：✅ 保留

**经验教训**：
- timeout 确实需要增加
- 但响应时间变长，需要权衡
- 下次可以尝试 450s 中间值
```

## 📈 评估指标

### 核心指标（必须追踪）
| 指标 | 目标值 | 计算方式 |
|------|--------|---------|
| **任务完成率** | ≥90% | 成功任务数/总任务数 |
| **timeout 错误** | ≤1 次/每 10 任务 | timeout 次数/完成任务数 |
| **平均响应时间** | ≤180s | 总耗时/任务数 |

### 辅助指标（参考）
| 指标 | 说明 |
|------|------|
| Agent 活跃度 | 活跃 Agent 数/总 Agent 数 |
| 消息成功率 | sessions_send 成功数/总数 |
| 心跳稳定性 | 心跳失败次数 |

## 🚀 实验方向建议

### 超时优化（配置参数，非时间驱动）
- [ ] 测试 timeout 配置：低 vs 中 vs 高
- [ ] 测试 runTimeoutSeconds 配置：低 vs 中 vs 高
- [ ] 测试 poll timeout 配置：低 vs 中 vs 高
- [ ] 测试 browser timeout 配置：低 vs 中 vs 高

### 并发优化
- [ ] maxConcurrent：低 vs 中 vs 高
- [ ] subagents.maxConcurrent：低 vs 中 vs 高
- [ ] 心跳并发：1 vs 3 vs 5

### 通信优化
- [ ] sessions_send vs feishu_im_user_message
- [ ] 消息格式：JSON vs Markdown
- [ ] 重试次数：0 vs 1 vs 3
- [ ] 超时重连：enabled vs disabled

### 模型路由优化
- [ ] primary 模型：Qwen vs GLM vs GPT
- [ ] fallback 链长度：1 vs 3 vs 5
- [ ] 模型切换阈值：连续 2 次失败 vs 3 次

### 心跳配置优化（配置参数，非时间驱动）
- [ ] 心跳触发条件：消息数 vs 任务数 vs 波次
- [ ] 活跃时段：工作时间 vs 全天
- [ ] lightContext：true vs false
- [ ] isolatedSession：true vs false

---

## ⚠️ 注意事项

### 1. 控制变量
- 每次只改一个参数
- 其他配置保持不变
- 确保数据可归因

### 2. 测试环境
- 使用测试账号（不影响生产）
- 记录测试时间段
- 避开高峰期（数据异常）

### 3. 回滚机制
- 改动前备份配置
- 失败时立即回滚
- 记录回滚原因

### 4. 安全底线
- 不突破权限限制
- 不修改安全配置
- 不影响数据安全
