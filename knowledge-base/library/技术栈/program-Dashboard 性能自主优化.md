---

## 🎯 目标

**改进 SQL 查询/索引配置以降低查询延迟**

---

## 📋 核心规则

### 1. 每次只改一个东西
- ✅ 索引（新增/删除/修改）
- ✅ SQL 查询语句
- ✅ 数据库配置（work_mem/shared_buffers）
- ✅ 连接池配置
- ✅ 查询缓存配置

**禁止**：一次改多个东西（无法归因）

---

### 2. 记录改动原因
每次改动前必须写：
```markdown
## 实验记录 #001

**改动内容**：为 tasks 表添加 (status, created_at) 复合索引

**改动原因**：
- 09:00 指标分析查询慢（全表扫描）
- PostgreSQL 索引优化指南建议复合索引列顺序
- 预期查询延迟降低 50%+

**预期效果**：查询延迟从 2s 降低到 1s 以内
```

---

### 3. 跑基准测试（进度驱动）
- 运行基准测试脚本
- **执行 100 次查询**（固定样本量）
- 记录 P50/P90/P99 延迟

---

### 4. 数据驱动决策
| 指标 | 决策 |
|------|------|
| P99 延迟降低≥50% | ✅ 保留，继续优化 |
| P99 延迟降低 30-50% | 🟡 保留，观察长期 |
| P99 延迟变化<30% | ❌ 回滚，换方向 |
| P99 延迟增加 | ❌ 立即回滚，记录教训 |

---

## 🔄 实验循环（进度驱动）

```
1. 读取当前配置（索引/SQL/配置）
   ↓
2. 分析数据（慢查询日志/性能指标）
   ↓
3. 提出假设（"如果加 X 索引，延迟会降低"）
   ↓
4. 修改配置（只改一个东西）
   ↓
5. 跑基准测试（执行 100 次查询）
   ↓
6. 收集指标（P50/P90/P99）
   ↓
7. 决策（保留/回滚）
   ↓
8. 记录实验日志
   ↓
9. 重复循环
```

## 📊 实验日志模板

```markdown
## 实验 #001

**日期**：2026-04-08

**改动内容**：为 tasks 表添加 (status, created_at) 复合索引

**改动原因**：
- 09:00 指标分析查询慢（全表扫描）
- PostgreSQL 索引优化指南建议复合索引列顺序

**预期效果**：查询延迟从 2s 降低到 1s 以内

**实际效果**：
- P50 延迟：1.2s → 0.3s（-75%）✅
- P90 延迟：2.0s → 0.5s（-75%）✅
- P99 延迟：3.5s → 0.8s（-77%）✅
- 全表扫描：10 次 → 0 次（-100%）✅

**决策**：✅ 保留

**经验教训**：
- 复合索引确实有效
- 列顺序很重要（status 在前）
- 下次可以尝试覆盖索引
```

## 📈 评估指标

### 核心指标（必须追踪）
| 指标 | 目标值 | 计算方式 |
|------|--------|---------|
| **P50 延迟** | ≤500ms | 50% 查询的延迟 |
| **P90 延迟** | ≤1s | 90% 查询的延迟 |
| **P99 延迟** | ≤2s | 99% 查询的延迟 |

### 辅助指标（参考）
| 指标 | 说明 |
|------|------|
| 全表扫描次数 | EXPLAIN 分析 |
| 索引命中率 | 命中索引次数/总查询数 |
| 查询吞吐量 | 查询数/秒 |

## 🚀 实验方向建议

### 索引优化
- [ ] 单列索引 vs 复合索引
- [ ] 复合索引列顺序（高选择性在前）
- [ ] 覆盖索引（包含所有查询字段）
- [ ] 部分索引（只索引符合条件的行）
- [ ] 表达式索引（索引计算结果）

### SQL 查询优化
- [ ] SELECT * vs SELECT 具体字段
- [ ] JOIN 顺序优化
- [ ] 子查询 vs JOIN
- [ ] LIMIT 优化（分页查询）
- [ ] 批量查询 vs 单次查询

### 数据库配置优化
- [ ] work_mem：4MB vs 16MB vs 64MB
- [ ] shared_buffers：256MB vs 512MB vs 1GB
- [ ] effective_cache_size：1GB vs 2GB vs 4GB
- [ ] random_page_cost：4.0 vs 1.1（SSD）

### 连接池优化
- [ ] 连接数：10 vs 20 vs 50
- [ ] 空闲超时：60s vs 300s vs 600s
- [ ] 连接复用：enabled vs disabled

## ⚠️ 注意事项

### 1. 12 个索引失效陷阱（来自玄机学习汇报）
- [ ] 复合索引列顺序错误
- [ ] 部分索引不匹配
- [ ] 函数/表达式导致索引失效
- [ ] OR 条件导致索引失效
- [ ] LIKE '%xxx' 前缀通配符
- [ ] 类型隐式转换
- [ ] NOT/<> 否定条件
- [ ] IS NULL/IS NOT NULL
- [ ] 范围查询后的列
- [ ] UNION vs UNION ALL
- [ ] 子查询未优化
- [ ] 统计信息过期

### 2. 控制变量
- 每次只改一个东西
- 其他配置保持不变
- 确保数据可归因

### 3. 测试环境
- 使用测试数据库（不影响生产）
- 使用相同数据量
- 记录测试时间段

### 4. 回滚机制
- 改动前备份配置
- 失败时立即回滚
- 记录回滚原因
