### CSS @property 渐变边框动画 🆕
- **来源**：霓裳 | 2026-04-11
- **核心能力**：`conic-gradient` + `@property` + `::before` 实现旋转渐变边框，纯 CSS 动画
- **实现原理**：
  1. `@property` 注册 CSS 自定义属性为角度类型（`<angle>`）
  2. `conic-gradient` 使用自定义属性作为起始角度
  3. `@keyframes` 动画旋转角度值
  4. `::before` 伪元素做渐变边框，`mask` 或 `clip-path` 裁出内部区域
- **示例代码**：
  ```css
  @property --angle {
    syntax: "<angle>";
    initial-value: 0deg;
    inherits: false;
  }
  .card::before {
    content: "";
    position: absolute;
    inset: 0;
    background: conic-gradient(from var(--angle), #ff6b6b, #4ecdc4, #45b7d1, #96ceb4, #ff6b6b);
    animation: spin 3s linear infinite;
    z-index: -1;
    border-radius: inherit;
  }
  @keyframes spin {
    to { --angle: 360deg; }
  }
  ```
- **应用场景**：
  - Nova ChannelCard hover 效果升级
  - 任何需要视觉吸引力的卡片组件
  - CTA 按钮渐变边框
- **价值**：hover 视觉吸引力大幅提升，纯 CSS 零 JS 开销
- **适用**：霓裳、Nova 项目、所有前端组件

### AI 产品路线图策略（三地平线框架）🆕
- **来源**：学习汇报 | 2026-04-11
- **核心认知**：AI 路线图≠传统路线图

  - **数据依赖**：需要数据准备/清洗/标注
  - **非线性提升**：模型效果不是线性增长，可能突然跃升或 plateau
  - **模型退化**：数据漂移导致模型效果下降，需要持续维护
- **三地平线框架**：
  1. **Committed（已验证，1-2 月）**：技术可行，有数据，明确交付物
  2. **Planned（有方案，2-3 月）**：有初步方案，需验证可行性
  3. **Exploring（探索中，3-6 月）**：研究方向，不确定能否交付
- **RICE-AI 适配**：
  - **Reach**：影响用户数
  - **Impact**：业务影响程度
  - **Confidence = 四维相乘**：数据可用性 × 模型能力 × 工程可行性 × 用户接受度
  - **Effort**：开发成本
- **时间盒管理**：
  - 先 1-2 周研究时间盒 → 验证可行性 → 再决定是否执行
  - 不是盲目开发，研究也是交付物
- **MVAI（Minimum Viable AI）**：
  - 最简 AI 版本，快速验证核心价值
  - 不是完整模型，是验证假设的最小实验
- **持续维护预算**：
  - AI 产品需要持续的数据/模型维护，不是一次性交付
  - 路线图预留 20% 维护容量
- **应用场景**：

### AI-Native 开发者生产力基准 2026 🆕
- **来源**：学习汇报 | 2026-04-10
- **核心发现**：传统 LOC/PRs 指标在 AI 时代失效，需用新基准衡量
- **新指标体系**：
  1. **CAT（Complexity-Adjusted Throughput，复杂度调整吞吐量）**：
     - 不是数代码行数，而是考虑代码复杂度的真实产出
     - 公式：产出功能点数 / 复杂度系数
  2. **Code Turnover Rate（代码周转率）**：
     - AI 生成代码被修改/删除的频率
     - 高周转率 = AI 生成代码质量差，需要大量人工修改
     - 低周转率 = AI 生成代码可直接使用
  3. **Review Time per PR**：AI 时代审查时间可能增加（更多代码要审）
  4. **Defect Escape Rate**：AI 生成代码的缺陷逃逸率
- **传统指标失效原因**：
  - LOC：AI 一键生成千行代码，但不一定有实际价值
  - PRs：AI 可能一天开 10 个 PR，但质量参差不齐
  - 提交频率：AI 可以自动提交，不代表真实生产力
- **应用场景**：
  - 评估 AI 编程工具效能（Copilot/Claude Code/Codex）
  - 给工程团队建议：用什么指标衡量 AI 时代的生产力
  - 技术选型：AI 工具 ROI 评估
- **价值**：用量化基准替代模糊的"AI 提高效率"说法
- **适用**：天工（工程效能）、玄机（开发度量）、年年（AI 工具评估）
- **优先级**：🔴 高
- **状态**：✅ 新录入

### 风险驱动测试（Risk-Driven Testing）🆕
- **来源**：学习汇报 | 2026-04-10
- **核心原则**：时间不够时要按风险优先级而不是平均用力
- **实施方法**：

  3. **按风险分配精力**：80% 时间测 P0+P1，20% 测 P2+P3

- **映射到测试**：
  - ✅ 先测核心业务流（下单/支付/登录）
  - ✅ 再测边缘场景
- **价值**：高风险问题命中率更高，低价值扫描更少

### 两两组合测试（Pairwise Testing）🆕
- **来源**：学习汇报 | 2026-04-10
- **核心原则**：多变量场景先做 pairwise 覆盖，比拍脑袋抽样更稳也更省
- **为什么有效**：
  - 研究表明 70-90% Bug 由 2 因素交互引起
  - 全组合测试：3 个变量×4 个值 = 64 种组合
  - Pairwise 覆盖：只需 16 种组合 → 节省 75%
- **实施方法**：
  1. 识别所有变量（权限/地区/浏览器/订单状态等）
  2. 列出每个变量的取值
  3. 生成 pairwise 组合矩阵（保证每对变量的每个值组合至少出现一次）
  4. 按矩阵执行测试
- **应用场景**：
  - 权限×角色×操作组合
  - 浏览器×设备×功能组合
  - 订单状态×支付方式×地区组合
  - 审稿：角色×状态×场景多变量交互排查
- **工具推荐**：PICT（微软）、AllPairs、正交表
- **价值**：覆盖效率更高，组合爆炸更可控
- **适用**：鉴微（组合测试）、玄机（多变量功能开发）、年年（审稿多变量排查）
- **优先级**：🔴 高
- **状态**：✅ 新录入

### 决策表测试（Decision Table Testing）🆕
- **来源**：学习汇报 | 2026-04-10
- **核心原则**：复杂规则场景要把条件和动作摊成决策表，不然很容易漏组合
- **决策表结构**：
  - **条件桩**：所有影响决策的条件（如"是否会员"、"订单金额>100"、"使用优惠券"）
  - **动作桩**：所有可能的结果（如"打 9 折"、"免运费"、"送积分"）
  - **规则列**：每种条件组合对应的动作
- **为什么需要**：
  - ❌ 口头规则："会员满 100 减 20，非会员满 200 减 30，用券不能叠加..."→ 容易漏
  - ✅ 决策表：条件×动作矩阵→ 一目了然，漏组合立即可见
- **应用场景**：
  - 优惠券叠加规则（会员×金额×券类型×活动时间）
  - 权限矩阵（角色×资源×操作×数据范围）
  - 审批流程（金额×部门×紧急程度→ 审批人）
  - 审稿：人物行为合理性推导（角色性格×当前情境×目标→ 行为是否合理）
- **价值**：规则覆盖更完整，漏分支和冲突判断更少
- **适用**：鉴微（规则测试）、玄机（规则引擎开发）、望舒（产品规则定义）、年年（审稿逻辑推导）
- **优先级**：🔴 高
- **状态**：✅ 新录入

### 性质驱动测试（Property-Based Testing）🆕
- **来源**：学习汇报 | 2026-04-10
- **核心原则**：先定义系统绝不能违反的不变量，再让大量输入去找反例，比堆例子更能打到深层 bug
- **什么是不变量（Invariant）**：
  - 金额不能为负（无论什么场景）
  - 库存不能 < 0（无论并发多高）
  - 幂等操作重复执行结果一致
  - 状态转移必须符合规则（不能从"待支付"直接跳到"已完成"）
- **测试方法**：
  1. 定义不变量（系统必须满足的性质）
  2. 生成大量随机输入
  3. 自动找反例（违反不变例的输入）
  4. 缩小反例到最小复现
- **工具推荐**：Hypothesis（Python）、fast-check（JS/TS）、PropEr（Erlang）
- **应用场景**：
  - 金额计算测试（无论怎么算，总额 = 单价×数量 - 优惠）
  - 库存并发测试（无论多少并发，库存不能 < 0）
  - 幂等性测试（重复提交，结果一致）
  - 状态一致性测试（状态转移必须符合规则）
  - 审稿：人物性格不变量（胆小鬼不会突然勇敢）、世界规则不变量（魔法消耗必须有代价）
- **价值**：复杂逻辑问题更容易暴露，定位最小复现更快
- **适用**：鉴微（复杂逻辑测试）、玄机（业务逻辑开发）、年年（审稿不变量检查）
- **优先级**：🔴 高

### 变异测试（Mutation Testing）🆕
- **来源**：学习汇报 | 2026-04-10
- **核心原则**：覆盖率 100% ≠ 测试有效，要用变异测试量化测试到底能不能抓到问题
- **什么是变异测试**：
  - 故意修改代码（制造"变异体"），看测试是否失败
  - 如果测试通过了 → 说明测试没抓到这个 bug → 测试无效
  - 如果测试失败了 → 说明测试有效 → "杀死"了这个变异体
- **Mutation Score** = 杀死的变异数 / 总变异数 × 100%
- **常见变异操作**：
  - 条件反转：`if (x > 0)` → `if (x <= 0)`
  - 边界值修改：`i < len` → `i <= len`
  - 运算符替换：`a + b` → `a - b`
  - 返回值修改：`return true` → `return false`
  - 删除语句：删除关键断言或计算
- **为什么需要**：
  - ❌ 行覆盖率 100% → 代码执行了但断言可能太弱
  - ✅ Mutation Score 80% → 80% 的 bug 能被测试抓到
- **工具推荐**：Stryker（JS/TS）、mutmut（Python）、Pitest（Java）
- **应用场景**：
  - 关键模块测试质量验证（支付/权限/库存）
  - 测试用例优化（找出杀不死变异的测试盲区）
  - 审稿对抗性验证：故意改情节/人设/时间线，看审稿流程能否发现
- **价值**：测试有效性从主观判断变成 mutation score 可量化指标
- **适用**：鉴微（测试质量度量）、玄机（关键模块开发）、年年（审稿对抗性验证）
- **优先级**：🔴 高
- **状态**：✅ 新录入

### 不稳定测试预防（Flaky Test Prevention）🆕
- **来源**：学习汇报 | 2026-04-10
- **核心原则**：flaky 测试不是小毛病，是 CI 可信度的最大杀手
- **关键数据**：Google 发现 84% 的 pass-to-fail 是 flaky 不是真 bug
- **什么是 Flaky 测试**：
  - 同一代码，有时通过有时失败
  - 不可复现 → 开发者不信任测试结果
  - 最终结果：所有测试都被忽略 → CI 形同虚设
- **常见原因**：
  - 时序竞争（async 操作未等待）
  - 随机性（依赖随机数/时间戳）
  - 环境依赖（网络/数据库/文件系统状态）
  - 全局状态污染（测试间相互影响）
- **预防策略**：
  1. 隔离：每个测试独立环境，不共享状态
  2. 确定性：mock 所有随机性/时间/网络
  3. 等待：用显式等待替代 sleep
  4. 重试：flaky 测试自动重试 + 标记
  5. 监控：追踪 flaky 率，超过阈值阻断 CI
- **应用场景**：
  - 测试稳定性设计
  - CI 结果可信度保障
  - 审稿输出一致性（同样内容每次审查结果应一致）
- **价值**：减少不可复现问题，提升测试和审查结果的可信度
- **适用**：鉴微（测试质量）、玄机（CI/CD）、年年（审稿一致性）
- **优先级**：🔴 高
- **状态**：✅ 新录入

### API 协议选型决策框架 2026 🆕
- **来源**：学习汇报 | 2026-04-11
- **核心原则**：按通信边界选型，不是"哪个协议最好"
- **四类边界匹配**：
  1. **Public API（对外公开）→ REST**：
     - 原因：简单通用、缓存友好、浏览器原生支持
     - 场景：第三方集成、开放平台、移动端
  2. **BFF（Backend for Frontend）→ GraphQL**：
     - 原因：前端按需查询、减少过度获取/获取不足
     - 场景：Web 前端、移动端、多端适配
  3. **S2S（Service-to-Service）→ gRPC**：
     - 原因：高性能、Protobuf 序列化、强类型契约
     - 场景：微服务内部通信、高吞吐低延迟
  4. **AI Agent → MCP（Model Context Protocol）**：
     - 原因：AI Agent 标准通信协议、工具发现/调用标准化
     - 场景：AI Agent 与外部工具/数据源通信
- **常见错误**：
  - ❌ 一刀切：全项目用同一个协议
  - ❌ 过度设计：内部服务用 REST + JSON（性能差）
  - ❌ 忽略边界：对外用 gRPC（客户端支持差）
- **应用场景**：
  - 微服务架构设计
  - AI Agent 项目通信选型
  - 新服务协议决策
- **价值**：不再一刀切推荐协议，而是按边界匹配
- **适用**：天工（架构设计）、玄机（微服务开发）、年年（多 Agent 协调）
- **优先级**：🔴 高
- **状态**：✅ 新录入

### 微服务分层测试策略 🆕
- **来源**：学习汇报 | 2026-04-11

  - **E2E-heavy = 分布式单体反模式**：太多端到端测试导致慢反馈循环，微服务变成了"分布式单体"
  - **PR gate 只跑 unit+contract**：快速反馈，不阻塞合并
  - **Contract Testing 不可跳过**：微服务接口一致性保障的核心
- **分层测试金字塔**：
  1. **Unit Tests（底层，最多）**：单个函数/类，毫秒级反馈
  2. **Contract Tests（中间层）**：服务间接口契约验证（Pact/OpenAPI）
  3. **Integration Tests（少量）**：服务 + 依赖（DB/消息队列）
  4. **E2E Tests（最少，顶层）**：关键用户流程，不是每个场景都跑
- **PR Gate 策略**：
  - ✅ 必跑：Unit + Contract（快速反馈，<5 分钟）
  - ✅ 异步跑：Integration（不阻塞合并）
  - ✅ 定时跑：E2E（夜间/部署前）
  - ❌ 不跑：完整 E2E 套件在 PR gate 里
- **应用场景**：
  - 微服务项目 CI/CD 设计
  - 避免慢反馈循环
  - 分层测试替代 E2E 一把抓
- **价值**：避免慢反馈循环，用分层测试替代 E2E 一把抓
- **适用**：鉴微（测试策略）、天工（CI/CD 架构）、玄机（微服务开发）

### BDD 行为驱动开发（Behavior-Driven Development）🆕
- **来源**：学习汇报 | 2026-04-11

- **三大支柱**：
  1. **Three Amigos 会话**：产品 + 开发 + 测试三方对齐，消除需求模糊

  3. **活文档（Living Documentation）**：
     - 测试即文档，自动更新，不是维护过时文档

    - Given：读者读到第 X 章
    - When：看到主角揭露真相
    - Then：读者感到震惊 + 想看下一章
- **应用场景**：
  - 需求评审用 Three Amigos 消除模糊
  - 审稿中用"章节行为定义"确认读者体验
  - 验收标准用 Gherkin 格式
- **价值**：减少"做对了但做错了东西"的返工
- **适用**：鉴微（验收标准）、望舒（需求评审）、惊鸿（章节行为定义）、年年（质量对齐）

### 缺陷根因分析（5 Whys + 帕累托分析）🆕
- **来源**：学习汇报 | 2026-04-11
- **核心认知**：修 bug 不够，要找到为什么 bug 会发生并防止再出现
- **5 Whys 分析法**：
  - 连续问 5 次"为什么"，追问到流程/设计层
  - 例：为什么有 bug？→ 因为没测到 → 为什么没测到？→ 因为没写测试 → 为什么没写？→ 因为 TDD 没执行 → 为什么没执行？→ 因为流程没强制 → **根因：流程缺陷**
  - 不是停在"开发者粗心"这种表面原因
- **帕累托分析（80/20 法则）**：
  - 80% 的缺陷来自 20% 的关键原因
  - 找到这 20% 集中解决，效率最大化
  - 例：80% 的 bug 来自 3 个模块 → 重点重构这 3 个模块
- **创新应用（内容创作）**：
  - 审稿中重复写作问题追根因：
    - 为什么写不好？→ 因为 prompt 不清晰 → 为什么不清？→ 因为没定义标准 → **根因：缺乏写作规范**
  - 分类：prompt 问题 / skill 问题 / 流程问题
- **应用场景**：
  - 测试中高频缺陷 5 Whys 分析
  - 审稿中重复写作问题追根因
  - 质量改进中帕累托分析
- **价值**：同类问题重复出现率下降，从发现→修复升级为发现→根因→预防
- **适用**：鉴微（缺陷分析）、惊鸿（写作问题根因）、年年（质量改进）
- **优先级**：🔴 高
- **状态**：✅ 新录入

### QA 指标与 KPI 体系 🆕
- **来源**：学习汇报 | 2026-04-11
- **核心认知**：质量指标≠KPI
  - **指标（Metrics）**：原始数据（缺陷数量/测试覆盖率/通过率）
  - **KPI（Key Performance Indicators）**：关键决策指标（缺陷检测率/逃逸率/MTTR）
  - 指标是"是什么"，KPI 是"意味着什么"
- **关键原则**：
  - **成熟团队跟踪少量 KPI**：3-5 个关键指标，不是 20+ 个原始数据n  - **关注趋势而非绝对数值**：趋势比单次值更有决策意义n  - **KPI 必须支持决策**：不能支持决策的指标不要跟踪n- **推荐 KPI**：n  1. **缺陷检测率（Defect Detection Rate）**：测试阶段发现的缺陷占比n  2. **缺陷逃逸率（Defect Escape Rate）**：生产环境发现的缺陷占比n  3. **MTTR（Mean Time To Recovery）**：平均恢复时间n  4. **测试覆盖率趋势**：不是绝对值，是上升/下降趋势n  5. **缺陷修复周期**：从发现到修复的平均时间n- **创新应用（内容创作）**：n  - 审稿报告区分发现数量和质量影响程度n  - 不是"发现 10 个问题"，而是"3 个高影响 +7 个低影响"n- **应用场景**：n  - 测试报告区分指标数据和关键 KPIn  - 质量报告关注缺陷检测率和逃逸率n  - 审稿报告区分发现数量和质量影响程度n- **价值**：质量报告更有针对性，用数据支持决策而非堆砌数字n- **适用**：鉴微（测试报告）、年年（质量报告）、惊鸿（审稿报告）n- **优先级**：🟡 中n- **状态**：✅ 新录入

- **来源**：学习汇报 | 2026-04-10

### 创意问题解决 CPS + 逆向头脑风暴 🆕
- **来源**：学习汇报 | 2026-04-10
- **Reverse Brainstorming（逆向头脑风暴）**：
  1. 先想"怎么让问题更糟？"（不是怎么解决）

### 因果回路图（Causal Loop Diagram, CLD）🆕
- **来源**：学习汇报 | 2026-04-10
- **核心概念**：
  - **增强回路（Reinforcing Loop）**：正向反馈循环（如病毒传播：分享→更多人看→更多人分享）
  - **调节回路（Balancing Loop）**：负向反馈循环（如内容疲劳：看太多→兴趣下降→减少观看）
  - **延迟（Delay）**：因果之间的时间差
- **分析步骤**：
  1. 识别关键变量（如内容质量、分享率、观众留存）
  2. 绘制变量间因果关系（+/-）
  3. 识别回路类型（增强/调节）
  4. 找到杠杆点（小改变→大影响）
- **应用场景**：
  - 内容传播机制分析
  - 用户增长/留存分析
  - 产品功能影响分析
- **价值**：从线性思维升级为系统思维，找到传播杠杆点
- **适用**：灵犀（创意策划）、望舒（产品分析）、年年（策略分析）

### 问题重构 Framing/Reframing 🆕
- **来源**：学习汇报 | 2026-04-10
- **核心原则**：创意质量常取决于是否先问对问题，而不是更快给答案
- **Framing（框架设定）**：
  - ❌ 错误框架："怎么让这个活动更炸？"→ 方向模糊，脑暴发散
  - ✅ 正确框架："用户在这个场景下真正未被满足的情绪是什么？"→ 精准定位
- **Reframing（重构问题）**：
  - "怎么提高转化率？" → "用户为什么不愿意行动？"
  - "怎么让更多人参与？" → "什么让用户觉得参与有价值？"
  - "怎么做得更好？" → "用户真正在意的是什么？"
- **重构步骤**：
  1. 识别原始问题中的假设
  2. 从用户/受众视角重新表述
  3. 聚焦情绪/需求，不是手段
  4. 验证新框架是否能导向更精准的创意
- **应用场景**：
  - 创意策划前先重构问题
  - 需求分析时深挖真实需求
  - 脑暴前确保问对问题
- **价值**：减少无效脑暴，提高创意命中率
- **适用**：灵犀（创意策划）、望舒（需求分析）、年年（任务拆解）
- **优先级**：🔴 高
- **状态**：✅ 新录入

### Implementation Intentions（执行意图）🆕
- **来源**：学习汇报 | 2026-04-10
- **核心公式**：When（何时）+ Where（何地）+ What（做什么）
  - ❌ 模糊目标："我要多运动"

- **关键洞察**：
  - 动机演讲对行为无影响 → 动机≠行动
  - 执行意图创建"触发条件→行为"的自动化链条
  - 不需要意志力，到时间自动触发
- **应用场景**：
  - 心跳流程本身就是执行意图：When（心跳触发）+ Where（HEARTBEAT.md）+ What（执行任务）
  - 给主人建议时用执行意图格式，不是模糊的"你应该..."

- **auto-instrumentation**：Node.js 微服务自动传播 W3C Trace Context

### 分布式系统韧性模式 🆕
- **来源**：学习汇报 | 2026-04-10
- **核心发现**：
  - **Retry Amplification（重试放大）**：重试不是免费的，指数级放大流量导致级联故障
    - 数学机制：10 服务×100ms 延迟×3 次重试 = 30 倍流量放大
    - 级联效应：下游延迟→上游重试→全链路崩溃
  - **5 种韧性模式应用顺序**：
    1. Timeout（超时）→ 最快止损
    2. Retry（重试）→ 带 jitter 的指数退避
    3. Circuit Breaker（熔断器）→ 防止持续失败
    4. Rate Limiter（限流）→ 保护下游
    5. Bulkhead（隔离舱）→ 故障隔离
  - **explicit latency budget 框架**：
    - 每个服务有明确的延迟预算
    - 不是随意给 retry/timeout 建议
    - 基于端到端 SLO 反推每层预算
- **应用场景**：
  - 微服务架构设计
  - 故障分析和根因定位
  - 系统韧性评估
- **价值**：避免给出随意的 retry/timeout 建议，用 explicit latency budget 框架
- **适用**：天工（架构设计）、玄机（微服务开发）、年年（系统架构评审）
- **优先级**：🔴 高
- **状态**：✅ 新录入
